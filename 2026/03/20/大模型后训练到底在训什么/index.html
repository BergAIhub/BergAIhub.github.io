大模型后训练到底在训什么

预训练给了模型知识，后训练教它做人。这篇文章讲清楚 SFT、RLHF、DPO、RLVR 各自在干什么，LoRA 和它们是什么关系，以及一个更关键的判断：什么时候该训，什么时候根本不用训。

先说结论

后训练是一组技术的统称，解决一个核心问题：base model 像一个读完所有书但不会说人话的人，后训练把它变成能正常对话的助手。

但 2025 年之后更重要的判断是：大部分场景你根本不需要后训练。Prompt + RAG 能解决 80% 的问题。后训练只在格式强约束、小模型降本、数据敏感三个场景才值得投入。

基础知识

预训练 vs 后训练

预训练：几万亿 token 的互联网文本，训一个”下一个词预测器”。烧几千万美元，跑几个月。产出是 base model——什么都”知道”，但什么都不”会做”。

后训练：在 base model 上用少量高质量数据教它”怎么做事”。成本低得多，但决定最终体验。

类比：预训练是让人读完图书馆。后训练是教这个人怎么和别人说话。

Decoder-only 架构

GPT、Claude、Qwen、DeepSeek、LLaMA 全部是 Decoder-only Transformer。

核心机制四件套：

  • 输入 token 序列，预测下一个 token
  • Causal Mask 保证只看前面
  • KV Cache 加速推理
  • RoPE 做位置编码

后训练不改架构，只改权重。

对齐

后训练的目标叫”对齐”——让模型行为和人类期望一致。三层：有用、诚实、无害。所有后训练技术都在优化这三个目标的某个子集。

后训练的流水线

SFT 解决”会不会说人话”。RLHF/DPO 解决”说的话好不好”。两步是递进，不是替代。

四种方法

SFT

准备”问题-回答”对，让模型模仿。标准交叉熵损失。

解决的问题：让 base model 从”续写文本”变成”回答问题”。

局限：只能教模仿，不能教判断。模型学到表面模式，不是质量标准。

RLHF

三步：收集人类偏好标注 → 训 Reward Model → PPO 优化。

解决的问题：让模型不只是”能回答”，而是”回答得好”。

局限：Reward Model 有偏差、PPO 不稳定、工程复杂度极高（同时维护 4 个模型）。

DPO

跳过 Reward Model，直接用偏好对训模型。一个损失函数搞定。

和 RLHF 目标一样，工程简单 10 倍。代价是对数据质量要求更高，出问题不好排查。

2025 年的选择：小团队用 DPO，大厂用 RLHF。

RLVR

随 DeepSeek-R1 火起来。对有标准答案的任务（数学、代码），直接用”答案对不对”做奖励，不需要人类标注。用 GRPO 优化。

局限：只适用于可验证任务。开放式对话用不了。

容易想错的地方

“后训练就是 LoRA”

LoRA 不是后训练方法，是省显存的实现手段。后训练方法定义”训什么目标”，LoRA 定义”怎么省着训”。两者正交——你可以用 LoRA 做 SFT，也可以用 LoRA 做 DPO，也可以不用 LoRA 全参数训。

LoRA 的本质：每层加两个低秩小矩阵，训练参数从几十亿降到几百万。

“后训练是必须的”

最大的误解。GPT-4 / Claude / DeepSeek V3 / Qwen2.5 未微调就能做绝大部分下游任务。Prompt + RAG 覆盖 80% 需求。

“训练和推理用同一个工具”

训练用 PyTorch + PEFT + LLaMA-Factory。推理用 Ollama / vLLM。Ollama 不做训练。训好的模型要转 GGUF 格式才能上 Ollama。

底层约束

四个绕不开的代价：

数据：效果完全取决于数据质量。几千条高质量 > 几万条低质量。数据是最贵的成本。

灾难性遗忘：教它新能力，旧能力可能退化。固有代价。

评测困难：”变好了吗？”极难回答。自动指标只覆盖一部分，人工评测贵且主观。

运维成本：base 模型升级，你的微调就作废。Qwen2.5 → Qwen3，LoRA adapter 直接报废。

结论：后训练是高成本、高风险、高维护的操作。

知识注入的三条路径

把后训练放到统一框架里看：

  • 参数化（后训练）：知识压进权重。永久但不可逆。
  • 上下文化（Prompt / RAG）：知识放进 context window。动态可更新。
  • 工具化（Function Call / MCP）：知识外挂到工具。实时精确。

选哪条取决于知识特性：

  • 静态 + 极快响应 → 参数化
  • 动态 + 可追溯 → 上下文化
  • 实时 + 精确计算 → 工具化

这个框架叫 Context Engineering。不再纠结”要不要微调”，而是问”知识从哪条路径注入最经济”。

三个场景验证

心理咨询机器人：风格稳定性 → LoRA 微调；咨询知识 → RAG；危机干预 → Prompt。三条路径组合。

行业研究 Agent：通用推理 → 强 base model；行业知识 → RAG；搜索计算 → Function Call；报告格式 → Prompt。完全不需要后训练。

代码补全工具：通用代码能力 → 不训；项目上下文 → RAG；风格约定 → Prompt。只有极致延迟要求时才考虑小模型微调。

判断方法

Step 1：强 base model + 好 Prompt 能做吗？能 → 不训。

Step 2：瓶颈是什么？知识不够加 RAG；格式不稳定考虑 SFT；推理不够考虑 DPO；延迟太高考虑小模型 LoRA。

Step 3：有高质量数据吗？没有 → 别训，数据不够训出来比不训更差。

Step 4：能承受运维吗？不能 → 用 Prompt + RAG，跟着 base 模型升级自动变强。

总结

后训练是”把知识压进权重”的手段。代价是数据成本、灾难性遗忘、运维负担。大部分场景的最优解是 Context Engineering。

能用 Prompt 解决的问题，用 Prompt 解决。这不是偷懒，是工程判断。
