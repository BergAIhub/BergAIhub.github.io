从 malloc 到物理页帧：操作系统内存管理的完整链路

这篇文章把操作系统内存管理从用户态的 malloc 一路追到物理页帧的分配，覆盖虚拟地址空间、页表、TLB、缺页中断、物理内存分配器（Buddy + Slab）、以及 COW 和 mmap 等关键机制。读完你会清楚一次内存访问在硬件和内核之间经历了什么。

你写了一行 malloc，背后发生了多少事

写 C 的时候，malloc(1024) 返回一个指针，你往里面写数据，程序跑得好好的。但这个指针指向的地址不是物理内存地址——它是虚拟地址。从这个虚拟地址到 DRAM 芯片上的某个位置，中间隔了五层机制：

  1. 用户态内存分配器（glibc ptmalloc2）管理虚拟地址空间的分配和回收
  2. 内核通过 brk/mmap 系统调用给进程映射虚拟地址区间
  3. MMU 硬件通过页表把虚拟地址翻译成物理地址
  4. TLB 缓存最近的翻译结果，避免每次都走页表
  5. 物理内存分配器（Buddy System）管理真实的物理页帧

五层各管各的事，缺一不可。

虚拟地址空间：每个进程看到的”假”内存

每个进程有自己独立的虚拟地址空间。64 位 Linux（4 级页表）下，每个进程能寻址 128 TB 虚拟空间（48 位有效地址，高位符号扩展）。

关键点：

  • 堆向上长，栈向下长，中间是 mmap 区域。
  • 用户空间和内核空间通过地址的高位区分。用户态代码访问内核地址会触发段错误。
  • 这整个布局都是”虚拟的”——物理内存里这些段不一定连续，甚至不一定存在（可能还没分配物理页）。

页表：虚拟地址到物理地址的翻译字典

CPU 执行指令时用的是虚拟地址，但 DRAM 只认物理地址。页表就是这两者之间的翻译字典。

分页的基本单位：4 KB 页

x86-64 的默认页大小是 4 KB（4096 字节）。虚拟地址空间和物理内存都被切成 4 KB 的块：虚拟的叫”页”（Page），物理的叫”页帧”（Page Frame）。

一个 48 位虚拟地址的拆分：

  • 高 36 位（4×9）用来索引四级页表，找到物理页帧号
  • 低 12 位是页内偏移，直接拼到物理页帧号后面

四级页表的查找过程

每一级页表是一个 4 KB 的页，包含 512 个 8 字节的条目（512 × 8 = 4096）。每个条目除了下一级表的物理地址，还包含权限位：Present、Read/Write、User/Supervisor、Accessed、Dirty 等。

为什么要四级而不是一级？ 如果用一张平坦的页表覆盖 48 位地址空间，需要 2^36 个条目 × 8 字节 = 512 GB——比大多数机器的物理内存还大。多级页表的好处是：没用到的虚拟地址区间，对应的页表页根本不需要分配。一个只用了几 MB 内存的进程，页表可能只占几十 KB。

TLB：让地址翻译不成为瓶颈

每次内存访问都走四级页表意味着 4 次额外的内存读取——这会让程序慢 5 倍。TLB（Translation Lookaside Buffer）是 CPU 内部的硬件缓存，存储最近使用的虚拟页→物理页帧映射。

典型的 TLB 规格（现代 x86 CPU）：

层级 条目数 命中延迟 覆盖范围（4KB 页）
L1 DTLB 64-72 1 cycle 256-288 KB
L1 ITLB 64-128 1 cycle 256-512 KB
L2 STLB 1024-2048 7-8 cycles 4-8 MB

TLB 命中时，地址翻译只需要 1 个时钟周期。TLB 未命中时，硬件 Page Walker 自动遍历页表（7-30 个周期，取决于页表页是否在 CPU 缓存中）。如果页表页也不在缓存里，需要访问主存，代价可达几百个周期。

TLB 刷新的代价：进程切换时，如果切换到不同地址空间的进程，TLB 中的旧映射全部失效。现代 CPU 用 PCID（Process Context ID）来标记 TLB 条目属于哪个进程，避免每次切换都全量刷新。

缺页中断：内存的”按需加载”

虚拟地址空间里的页不一定有对应的物理页帧。当 CPU 访问一个页表条目中 Present 位为 0 的页时，硬件触发缺页中断（Page Fault），控制权交给内核的缺页处理程序。

缺页分两种：

Minor Page Fault（次要缺页）：页的数据已经在内存中（比如在页缓存里），只需要更新页表映射。不涉及磁盘 IO，代价几微秒。

Major Page Fault（主要缺页）：页的数据不在内存中，需要从磁盘读取（比如 swap 分区或文件系统）。涉及磁盘 IO，代价几毫秒——比 minor fault 慢 1000 倍。

Demand Paging（按需分页）

Linux 的内存分配是”乐观”的：malloc 或 mmap 只是在虚拟地址空间里标记一段区间为”已分配”，但不立即分配物理页帧。只有当进程第一次读写这个地址时，触发缺页中断，内核才真正分配物理页并建立映射。

这意味着：

Copy-on-Write（写时复制）

fork() 创建子进程时，内核不复制父进程的所有物理页。而是让父子进程共享同一组物理页，但把页表条目标记为只读。当任何一方尝试写入时，触发缺页中断，内核才复制那一页，让两个进程各自拥有独立的副本。

这使得 fork() 的开销从”复制整个地址空间”降低到”复制页表”——对于一个占用 1 GB 内存的进程，fork 的时间从几百毫秒降到几毫秒。

物理内存分配：Buddy System 和 Slab

内核需要管理真实的物理页帧。这里有两层分配器：

Buddy System（伙伴系统）

Buddy System 管理物理页帧的分配和回收，以 2 的幂次方为单位。

分配逻辑：

  1. 需要 1 页（order 0），先看 order 0 的空闲链表有没有
  2. 没有 → 从 order 1 拿一个 2 页块，拆成两个 1 页块（互为”伙伴”），一个给请求者，一个放回 order 0 空闲链表
  3. order 1 也没有 → 继续向上找 order 2，拆分…

回收逻辑：

  1. 释放一个 order 0 的页
  2. 检查它的伙伴是否也空闲
  3. 如果是 → 合并成 order 1 的块，继续检查更高 order 的伙伴…

优点：分配和回收都是 O(1) 到 O(log n)，合并机制减少外部碎片。

缺点：只能分配 2 的幂次方大小，内部碎片不可避免（需要 5 页但只能分配 8 页）。

Slab Allocator

内核经常需要分配固定大小的小对象（task_struct、inode、dentry 等）。每次都走 Buddy System 太浪费——一个 1.7 KB 的 task_struct 要占一整个 4 KB 页。

Slab Allocator 在 Buddy System 之上建了一层缓存：

  1. 为每种对象类型创建一个 cache（比如 task_struct_cachep）
  2. 每个 cache 从 Buddy System 申请整页，切成固定大小的 slot
  3. 分配时从 cache 里取一个空闲 slot，释放时放回 cache
  4. 对象可以保持初始化状态，下次分配时不需要重新初始化

malloc 到底做了什么：用户态分配器的角色

malloc 不是系统调用——它是 C 库（glibc）里的函数。glibc 的 ptmalloc2 分配器在用户态管理一个内存池，只在池子不够用时才向内核要更多内存。

两种向内核要内存的方式：

方式 系统调用 适用场景 特点
扩展堆 brk / sbrk 小块分配（< 128 KB） 移动堆顶指针，连续地址空间
内存映射 mmap 大块分配（≥ 128 KB） 在 mmap 区域创建新映射，释放时可以直接归还内核

ptmalloc2 的内部结构：

  • Arena：每个线程有自己的 arena（或共享），减少锁竞争
  • Chunk：分配的基本单位，包含 size 字段和 prev_size 字段
  • Bins：空闲 chunk 按大小分类放入不同的 bin（fast bins、small bins、large bins、unsorted bin）
  • Top chunk：堆顶的大块，所有 bin 都找不到合适的 chunk 时从这里切

一次内存写入的完整路径

把所有层串起来，当你执行 p[0] = 'A' 时：

  1. CPU 拿到虚拟地址，查 TLB
  2. TLB 命中 → 直接得到物理地址，写入 L1 Cache → 完成（1-4 cycles）
  3. TLB 未命中 → 硬件 Page Walker 遍历四级页表
  4. 页表条目 Present=1 → 填充 TLB，重试访问 → 完成（7-30 cycles）
  5. 页表条目 Present=0 → 触发缺页中断
  6. 内核缺页处理程序判断：这个地址合法吗？（检查 VMA）
  7. 合法 → 调用 Buddy System 分配一个物理页帧
  8. 填零（安全要求，防止泄露其他进程的数据）
  9. 更新页表条目：填入物理页帧号，设置 Present=1
  10. 返回用户态，CPU 重试那条指令 → 这次 TLB 命中 → 完成

第一次访问一个新页的代价：几微秒（minor fault）。之后的访问：几纳秒（TLB 命中）。差距 1000 倍。

容易搞混的几个点

“malloc 分配的是物理内存”——错。 malloc 分配的是虚拟地址空间。物理内存是在第一次访问时通过缺页中断按需分配的。

“free 之后内存就还给操作系统了”——通常不是。 glibc 的 free 只是把 chunk 放回内部的空闲链表。只有通过 mmap 分配的大块才会在 free 时通过 munmap 归还内核。brk 分配的堆内存通常不会收缩。

“64 位系统有无限内存”——不是。 虚拟地址空间确实很大（128 TB），但物理内存是有限的。当物理内存不够时，内核会启动页面回收（LRU）或 OOM Killer。

“页越大越好”——看场景。 大页（2 MB Huge Page）减少 TLB 未命中，但增加内部碎片和内存浪费。数据库和 JVM 这类大内存应用适合大页；小进程用默认 4 KB 页更合适。

以后我看内存问题，先定位在哪一层

  1. 应用层：malloc/free 的使用是否正确？有没有内存泄漏？→ 用 Valgrind、AddressSanitizer
  2. 虚拟内存层：进程的虚拟地址空间是否耗尽？mmap 区域是否碎片化？→ 看 /proc/pid/maps
  3. 页表/TLB 层：TLB 未命中率是否过高？→ 用 perf stat 看 dTLB-load-misses
  4. 物理内存层：系统物理内存是否不足？是否在频繁 swap？→ 看 free -h、vmstat、/proc/meminfo
  5. 分配器层：Buddy System 碎片化？Slab 缓存膨胀？→ 看 /proc/buddyinfo、slabtop

每一层有自己的诊断工具和优化手段。不要在应用层的内存泄漏问题上去调内核参数，也不要在物理内存不足时只盯着 malloc 的返回值。
