文件编码与程序执行：从乱码问题理解底层原理

主要内容：从 Windows CMD 执行 UTF-8 编码的 BAT 文件乱码问题出发，深入探讨文件编码、编译器与解释器的区别，以及程序执行的底层原理。

一切的起点：一个乱码的 BAT 文件

我在 VS Code 里写了一个简单的 BAT 文件，内容就是几行中文输出。但在 CMD 里执行时，中文全部变成了乱码。为什么？

这个看似简单的问题，让我一路挖到了文件编码、编译器原理、程序执行流程等底层知识。

文件编码：字节与字符的映射规则

什么是编码？

文件在硬盘上存储的本质是一串字节。编码就是字节和字符的映射规则。

同样的字节，用不同的编码解读，会得到不同的字符。这就是乱码的根源。

UTF-8 vs GBK

这是两种不同的编码标准：

对比项 UTF-8 GBK
英文字符 1 字节 1 字节
中文字符 3 字节 2 字节
支持范围 全球所有语言 仅中文
国际化 标准编码 仅中国使用

关键点：英文部分两者完全相同，区别只在中文等多字节字符。

这就是为什么纯英文的 BAT 文件不会乱码，而有中文就会乱码。

编码由谁决定？

编码在”写”的那一刻就决定了。

写入者 默认编码
VS Code UTF-8
老版记事本 ANSI（GBK）
新版记事本 UTF-8
Write 工具 UTF-8

写入时选择什么编码，文件就按什么编码存储。之后读取时必须用相同的编码，否则就会乱码。

为什么 CMD 会乱码

核心原因：编码不匹配

流程如下：

CMD 是 Windows 的”老古董”，默认使用系统编码（中文 Windows 是 GBK）。它不会自动检测文件编码，所以 UTF-8 文件会被错误解读。

为什么 chcp 65001 没用？

我尝试在 BAT 文件开头加 chcp 65001（切换到 UTF-8 编码页），但仍然乱码。

原因是：CMD 在执行 chcp 65001 之前，就已经用 GBK 解析了整个文件。这行命令只影响输出时的编码，不影响文件解析。

解决方案

有两种方式：

  1. 用 GBK 编码保存 BAT 文件（让文件编码与 CMD 默认编码一致）
  2. 避免中文（英文在 UTF-8 和 GBK 中编码相同）

编译器 vs 解释器

在探索过程中，我理解了编译器和解释器的本质区别。

核心区别

编译器 解释器
方式 先翻译成机器码，再执行 边读边执行
产物 生成独立文件 不生成文件
执行速度 快（只编译一次） 慢（每次都要解释）
源码依赖 编译后不需要源码 必须有源码

类比理解

  • 编译器 = 提前翻译整本书，读者直接读译本
  • 解释器 = 同声传译，说一句翻一句
  • BAT = 直接读原文执行，不翻译

更底层：字节与位

字节的本质

硬盘上存储的一切，本质上都是 0 和 1。

UTF-8 变长编码

UTF-8 是变长编码，不同字符占用不同字节数：

字符类型 占用字节 例子
英文 1 字节 A → 41
中文 3 字节 你 → E4 BD A0
Emoji 4 字节 😀 → F0 9F 98 80

变长的好处：英文只占 1 字节，节省空间；中文需要更多字节来表示更多字符。

文件存储的本质

文件 = 一串字节，编码 = 字节和字符的映射表。

编码匹配原则

整个探索过程让我理解了一个核心原则：

写和读必须用同一种编码，否则乱码。

场景 结果
VS Code 写 UTF-8 + Python 读 UTF-8 ✅ 正常
VS Code 写 UTF-8 + CMD 读 GBK ❌ 乱码
记事本写 GBK + CMD 读 GBK ✅ 正常

总结

  • 编码是字节与字符的映射规则，写入时决定，读取时必须匹配
  • UTF-8 是现代主流编码，支持全球所有语言，英文兼容 ASCII
  • CMD 默认用 GBK 编码，所以 UTF-8 的 BAT 文件会乱码
  • 编译器一次性翻译，生成机器码文件，之后可直接执行
  • 解释器边读边执行，每次运行都需要源码
  • BAT 文件不翻译，直接执行 CMD 命令
  • 文件本质是一串字节，编码决定了如何解读这些字节

这次从一个乱码问题出发，一路挖到了文件存储的底层原理。技术学习就是这样，一个看似简单的问题，往往能串联起一整套知识体系。
