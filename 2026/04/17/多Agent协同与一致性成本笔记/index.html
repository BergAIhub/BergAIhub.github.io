多Agent协同与一致性成本笔记

多 Agent 不是“多开几个线程”，而是把一个大问题拆成多个可验证子问题，同时控制一致性成本。

先说结论

单 Agent 的上限来自上下文和串行时间。

多 Agent 的上限来自协调质量。

协同做得好，系统吞吐和可靠性都提升。

协同做不好，只会更快地产生互相冲突的错误。

先把基础知识补齐

多 Agent 至少包含三种角色：

  • 规划者：定义目标、分解任务、设定验收标准。
  • 执行者：在限定范围内完成子任务。
  • 裁决者：整合结果、处理冲突、决定最终输出。

这不是组织图，而是信息流图。

谁定义标准，谁就控制质量边界。

这个问题到底在讨论什么

我们讨论的不是“并行越多越快”。

真实问题是：并行化带来的速度收益，是否超过一致性维护成本。

一致性成本包括：

  • 状态同步成本：大家看到的任务状态是否一致。
  • 协议理解成本：接口定义、交付格式是否一致。
  • 冲突解决成本：结果冲突时如何裁决。

这些成本被低估时，多 Agent 几乎一定翻车。

从基础往外延伸

常见协同模式有三种：

  • 管道式：上游输出给下游，结构简单，延迟较高。
  • 星型式：中心调度多执行者，控制强，中心负载高。
  • 市场式：多执行者竞争方案，择优整合，质量高但成本大。

没有绝对最优模式，只有是否匹配任务形态。

高耦合任务适合星型或管道。

探索性任务适合市场式。

这个问题为什么容易想错

误区一是“把任务切得越细越好”。

切太细会导致交接成本高于执行成本。

误区二是“统一 Prompt 就能保证一致性”。

一致性来自共享状态和验收协议，不是来自措辞一致。

误区三是“冲突出现时让主 Agent 直觉选一个”。

没有证据比较机制，裁决就会退化成随机选择。

回到第一性原理

多 Agent 的底层约束是通信复杂度。

Agent 数量增加时，潜在通信路径近似按平方增长。

这意味着系统设计必须主动减少不必要通信。

否则新增 Agent 带来的不是能力增长，而是协作噪声增长。

真正的机制是什么

高质量协同系统通常抓三件事：

  1. 明确所有权：每个子任务有清晰责任边界和写入范围。
  2. 明确契约：输入输出结构固定，可机器校验。
  3. 明确仲裁：冲突时按证据质量、覆盖度、风险级别决策。

这里最关键的是“先契约后并行”。

没有契约的并行，本质是并发地制造不兼容结果。

用这个模型重新看问题

你会发现很多“多 Agent 很慢”的系统，慢在整合，不慢在执行。

典型症状是：

  • 执行者都交付了内容，但格式不统一，主控要二次翻译。
  • 子任务边界重叠，出现互相覆盖和回滚。
  • 冲突没有标准处理流程，反复人工介入。

这些都是架构问题，不是模型问题。

我现在怎么判断

我会用这五条判断是否该上多 Agent：

  1. 任务是否可拆成低耦合子问题。
  2. 每个子问题是否能定义清晰验收标准。
  3. 子任务之间是否能用结构化契约交接。
  4. 冲突是否可通过证据规则而非主观偏好裁决。
  5. 预期吞吐提升是否大于协调与整合成本。

满足前四条，再看第五条，通常决策就清晰了。

总结

多 Agent 的本质是“把认知工作流工程化”。

性能瓶颈不在推理本身，而在协同协议。

谁先把一致性成本设计进系统，谁就能稳定吃到并行红利。

深度思考

多 Agent 的核心矛盾是并行收益与一致性成本的博弈。团队常以为多开几个执行单元就能提速，结果发现整合时间远超执行时间。原因很简单：每增加一个 Agent，不只是增加算力，还增加沟通路径、协议摩擦和冲突仲裁成本。深层问题在于协同系统往往先并行后定约，导致输出格式不统一、责任边界重叠、结果互相覆盖。真正可扩展的协同，应先定义契约再开放并发：谁负责什么、交付结构是什么、冲突由什么证据裁决。再往下看，一致性成本不是坏事，它是系统复杂度的显性价格。你能测量它、管理它，就能控制架构增长；你忽略它，它就会在后期以延迟和事故的形式收回。多 Agent 不是免费午餐，而是一种需要纪律的组织工程。

最后给一个可执行标尺：把这套方法交给一个不在场的同事，他能否仅凭日志、配置和文档，在一小时内复盘决策并复现实验结果。如果做不到，说明系统仍依赖个人经验而不是工程能力。继续把隐性判断显式化，直到可测、可审计、可回滚。
