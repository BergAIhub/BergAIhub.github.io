Agent检索索引更新与一致性延迟

RAG 线上最常见的问题不是“检索不到”，而是“检索到的是昨天的世界”。

先说结论

检索系统的核心矛盾是新鲜度、准确率、成本三者拉扯。

索引更新策略如果不透明，Agent 会在“看似有证据”时给出过期结论。

先把基础知识补齐

索引更新通常有三种路径：

  • 全量重建：准确但慢、成本高。
  • 增量更新：快，但易产生碎片和漂移。
  • 混合策略：热点增量 + 周期全量校准。

这个问题到底在讨论什么

不是向量库选型问题。

核心是“知识变更到可检索之间的延迟是否可控、可观察、可解释”。

从基础往外延伸

工程上建议维护三个时间戳：

  • source_updated_at：源数据更新时间
  • indexed_at：被写入索引时间
  • served_at：被检索命中时间

有了这三者，才知道错在数据、索引还是缓存。

这个问题为什么容易想错

误区一：只关注召回率，不关注时效偏差。

误区二：认为增量更新天然更好。

增量如果没有删除同步和重排，会长期污染检索结果。

回到第一性原理

检索系统本质是延迟一致系统。

你无法消灭延迟，只能定义可接受延迟并围绕它做策略设计。

真正的机制是什么

可靠方案通常有：

  1. 变更检测队列：捕捉新增、修改、删除。
  2. 双索引切换：新索引验证后再切流。
  3. 热点优先刷新：高频文档优先更新。
  4. 结果标注时效：让 Agent 知道证据新旧。

我现在怎么判断

  1. 是否可量化索引滞后时间分布。
  2. 删除操作是否能及时反映到检索层。
  3. 是否存在索引版本回滚能力。
  4. 热点知识更新是否显著快于冷门知识。
  5. Agent 输出是否标注证据时间戳。

总结

检索质量不只取决于“找没找到”，还取决于“找到的是不是当前事实”。

把一致性延迟显式工程化，RAG 才能真正可信。

深度思考

检索系统的难点不是“能不能查到”，而是“查到的是不是此刻真实世界”。很多团队关注召回率，却忽略索引时效，导致 Agent 在事实已变更后仍引用旧证据，答案逻辑完美却结论过期。深层矛盾在于任何索引系统都是延迟一致：你不可能零延迟更新全量知识，只能在新鲜度、成本和稳定性之间取平衡。关键不是消灭延迟，而是让延迟可观测、可分层、可解释。成熟做法是给每条证据附带 source_updated_at、indexed_at、served_at 三个时间维度，让系统知道“旧在哪一层”。再进一步，热点数据应优先刷新，冷数据周期校准，并保留版本回滚能力。这样 Agent 才能在输出时识别证据新旧，必要时触发二次验证，避免把“过期真相”当当前事实。

最后给一个可执行标尺：把这套方法交给一个不在场的同事，他能否仅凭日志、配置和文档，在一小时内复盘决策并复现实验结果。如果做不到，说明系统仍依赖个人经验而不是工程能力。继续把隐性判断显式化，直到可测、可审计、可回滚。
