Agent鉴权边界与会话设计

Agent 接口一旦接入真实业务，最先出事故的通常不是推理，而是鉴权边界。

先说结论

Agent 系统里的鉴权不能只看“用户是否登录”。

你真正要控制的是三层身份：谁在提问、谁在代表谁执行、谁在为这次执行买单。

会话设计的核心也不是“记住聊天内容”，而是“在会话生命周期里保持权限一致性”。

先把基础知识补齐

传统 Web 的鉴权模型通常是用户登录后拿到 token，再访问 API。

Agent 场景会多出两类身份：

  • 代理身份：Agent 以系统身份调用工具。
  • 下游身份：工具需要特定租户、项目、数据域权限。

这导致一次调用里可能同时出现用户 token、服务 token、工具 token。

如果边界不清晰，最容易出现越权串用。

这个问题到底在讨论什么

我们讨论的不是 JWT 还是 Session。

我们讨论的是“权限上下文是否随任务流转而保持可验证”。

同一个用户在不同对话、不同项目、不同环境下，权限范围可能完全不同。

如果会话状态没有绑定权限快照，Agent 很容易把上一次上下文带到下一次执行。

从基础往外延伸

工程上可落地的做法是三段式鉴权：

  1. 请求入口鉴权：确认用户身份与租户归属。
  2. 任务执行鉴权：把可执行动作映射成最小权限能力集。
  3. 工具调用鉴权：每次调用下游服务时再做一次细粒度校验。

这三段必须相互独立。

入口通过，不等于执行期间全部合法。

这个问题为什么容易想错

误区一是“让 Agent 拿一个全能密钥更方便”。

方便的是开发，不是生产。

全能密钥会把任何提示注入或策略失误放大成全库风险。

误区二是“会话里记住用户就够了”。

记住用户不等于记住授权范围、策略版本和资源边界。

回到第一性原理

鉴权系统的底层约束是不可逆操作成本。

读取越权还能补救，写入越权和外发越权往往不可逆。

所以鉴权门槛应按动作风险分级，而不是统一一刀切。

真正的机制是什么

稳定方案通常具备：

  • 能力票据：把“能做什么”编码成短期可验证令牌。
  • 会话快照：保存当前权限上下文版本与过期时间。
  • 二次确认：高风险动作触发人工确认或多因子校验。
  • 全链路审计：记录谁在何时以何身份调用了什么资源。

我现在怎么判断

我会先看五点：

  1. 会话是否绑定租户、项目、环境三元组。
  2. 是否存在全能 token 被 Agent 直接持有。
  3. 高风险动作是否强制升阶鉴权。
  4. 权限策略变更后，历史会话是否自动失效或重验。
  5. 审计日志能否还原一次完整决策链。

总结

Agent 鉴权的关键不是“有没有登录”，而是“权限上下文能否在多步执行里不漂移”。

谁先把会话和能力边界工程化，谁就能把风险压在可控范围内。

深度思考

鉴权最难的地方不是拒绝未登录用户，而是防止“合法身份做了不合法动作”。Agent 系统天然跨多步、多工具、多上下文，身份边界一旦漂移，就会出现看起来都通过校验、结果却越权的数据事故。深层问题在于很多会话系统只记住了用户 ID，却没有记住当时的租户、项目、策略版本和风险级别。于是同一个用户在不同场景切换时，权限被无声继承。更稳的思路是把会话当“带版本的能力快照”，每次高风险动作都重新验证能力，而不是盲信历史状态。你要设计的不是一次登录成功，而是一条长期可证明的授权链。等你把授权链做成可审计、可失效、可回放，安全才从规则层走到工程层，组织才敢把更高权限交给 Agent 自动化。

最后给一个可执行标尺：把这套方法交给一个不在场的同事，他能否仅凭日志、配置和文档，在一小时内复盘决策并复现实验结果。如果做不到，说明系统仍依赖个人经验而不是工程能力。继续把隐性判断显式化，直到可测、可审计、可回滚。
