行业信息助手

Industry Information Assistant

基于多智能体协作的行业深度研究与智能问答平台

一个面向智慧交通、金融科技、医疗健康、能源电力等垂直行业的 AI 研究系统。
通过多智能体编排、对抗式审核与多源检索，把一句自然语言提问，变成一份带数据、带图表、带引用的深度调研报告。


📖 项目简介

行业信息助手 是一套围绕「深度研究 (Deep Research)」展开的全栈 AI 应用，目标是把传统行研工作中最耗时的”找资料 — 读资料 — 写报告”流程，压缩成一次自然语言对话。

不同于普通的 RAG 问答系统，本项目把研究过程拆解为 6 个专业 Agent 的协作流水线：从问题规划、深度搜索、数据抽取、可视化生成到报告撰写、对抗式审核，全过程由 LangGraph 状态机调度，所有中间产物（事实库、引用、图表、知识图谱）都暴露给前端，让用户看到 AI 怎么想、怎么查、怎么写，而不是一个不透明的黑盒。

这个项目解决什么问题

  • 行业研究效率低：人工撰写一份覆盖 4–5 个维度的行业报告往往需要 1–3 天，本系统在 2–5 分钟内交付初稿
  • AI 幻觉难以控制：通过 Critic 智能体的对抗式审核 + 补充搜索回环，把无依据陈述率压到可接受区间
  • 数据来源单一：同时打通公网搜索、本地知识库、关系数据库 (Text2SQL)、招投标 / 股票 / 政策外部 API 四类信源
  • 结果不可追溯：每条事实都带 source_url + credibility_score，每张图表都附生成代码，每份报告都可断点续接

核心价值

维度 体现
深度 多 Agent 协同 + 假设驱动 + 知识图谱构建
可信 对抗式审核 + 信源权威度加权 + 引用强制覆盖

✨ 核心特性

🤖 多智能体深度研究 (DeepResearch V2)

基于 LangGraph 状态机 编排 6 个专业 Agent，共享一份全局工作记忆 ResearchState：

  • ChiefArchitect (规划师) — 拆解问题、生成大纲、提取关键实体与研究假设
  • DeepScout (侦察员) — 并行执行子查询，从 Web 与本地知识库召回原始资料
  • DataAnalyst (数据分析师) — 从非结构化文本中抽取数据点、生成洞察
  • CodeWizard (代码极客) — 自动生成 ECharts 图表配置与可视化代码
  • LeadWriter (首席写手) — 串联章节草稿、统一引用风格、生成最终报告
  • CriticMaster (审核大师) — 对抗式审核，识别幻觉/逻辑漏洞/信源缺失

Critic 触发的智能路由：发现严重问题时进入 RE_RESEARCHING 重启搜索；仅文字问题则进入 REVISING 仅修订。最多迭代 3 轮，质量评分低于阈值则继续，达标即收敛。

🔄 ReAct 控制器 (Plan → Execute → Reflect → Synthesize)

独立于 V2 的另一条研究链路，实现经典 ReAct 范式 的优化版：先由 LLM 生成 3–6 个互补子查询，并行执行（asyncio.gather），再用反思阶段评估信息完整性，必要时触发 1–2 轮补充搜索。子查询智能性、并发性都比朴素链路强一个数量级。

🔍 双通道混合检索

  • 网络搜索通道：博查 API (Bocha)，覆盖时效性强的新闻、报告、市场数据
  • 本地知识库通道：DocMind 解析 PDF/Word/PPT → 切片 → DashScope text-embedding-v4 嵌入 (1024 维) → Milvus 向量库 → DashScope Rerank 重排
  • 两路结果在 state.facts 中合并，按 source_type 加权（official 1.0 / academic 0.9 / report 0.8 / news 0.6 / self_media 0.3）送入下游 Agent

🗄️ Text2SQL 自然语言查询

内置一套 可执行、可审计 的 Text2SQL 引擎：

  • 自带 industry_stats / company_data / policy_data 三张表的 Schema 描述，LLM 看着 schema 生成 SQL
  • 关键词级别的 SQL 安全校验：禁止 DROP / DELETE / UPDATE，禁止多语句、禁止 SQL 注释，强制以 SELECT 开头
  • 自动推荐可视化类型（line / bar / pie / table），与前端 ECharts 直接对接

📡 SSE 流式输出 + 检查点恢复

所有研究步骤通过 Server-Sent Events 实时推送（data: {...}\n\n）：

  • phase / thought / action / observation / search_result_item / chart / checkpoint_saved / research_complete 等事件类型，前端按类型渲染右侧研究面板
  • 每完成一个阶段（plan / research / analyze / write）即写入 research_checkpoints 表，包含完整 ResearchState + 前端友好的 ui_state（搜索结果、图表、参考文献）
  • 支持 断点恢复：刷新页面或会话重连时，前端从检查点重建研究面板，无信息损失

📊 行业数据采集层

基于 APScheduler 的定时任务系统，按行业关键词自动采集：

数据类型 来源 模型
行业新闻 博查 API + 网页正文抽取 IndustryNews
招投标公告 阿里云市场招投标 API BiddingInfo
股票行情 聚合数据 API (实时查询)
政策法规 自研 policy_search_service PolicyData

行业关键词在 config/industry_config.py 中配置，支持智慧交通、金融科技、医疗健康、能源电力四类预设，可扩展。

🧠 长期记忆与会话管理

  • 短期记忆：每个会话独立的 ChatSession + ChatMessage 历史
  • 长期记忆：LongTermMemory 表存储跨会话的用户偏好、关注实体、历史研究主题，按 user_id 索引
  • 用户登录后，新会话会自动加载相关长期记忆作为上下文前缀

📑 文档解析与知识库

  • DocMind 智能解析 — 阿里云文档智能服务，处理扫描件 PDF / 复杂版式 Word / PPT
  • 本地解析回退 — pdfplumber + python-docx + python-pptx，无 DocMind Key 时仍可工作
  • 多知识库隔离 — 每个 KnowledgeBase 对应一个 Milvus collection，按 kb_id 过滤搜索

📈 完整评测体系

独立目录 eval/ 设计了一套 不侵入业务代码 的生产级评测框架：

  • IndustryBench 自建数据集（11 领域 × 3 难度 = 35 题）+ HotpotQA-DR (50 题)
  • 三层指标：8 项规则指标 + 5 维 LLM-as-Judge + Bootstrap CI / Cohen’s d / Wilcoxon 显著性检验
  • 6 项消融实验：Critic 开关、检索通道、迭代轮次、模型规模、跨 benchmark、全量基线

🏗️ 系统架构

整体架构

DeepResearch V2 多智能体工作流

数据流要点：

  • 所有 Agent 读写同一份 ResearchState (TypedDict)，包含 outline / facts / data_points / charts / draft_sections / final_report / references / critic_feedback / quality_score 等 20+ 字段
  • 每完成一个阶段，UI 状态同步落入 research_checkpoints 表，前端可从任意中断点恢复
  • SSE 通道实时推送 phase / thought / action / observation / search_result_item / chart 事件，研究面板按类型渲染

🤖 核心模块详解

1. DeepResearch V2 多智能体引擎

入口：backend/app/service/deep_research_v2/

文件 职责
state.py 全局工作记忆 ResearchState 定义、ResearchPhase 状态机枚举
graph.py LangGraph StateGraph 构建 + 简化版异步执行循环 (asyncio.Queue 实时流)
service.py 对外服务封装，提供 research() (SSE 流式) 与 research_sync() (同步) 两种入口
agents/ 6 个 Agent 实现，均继承 BaseAgent，统一通过 client.chat.completions.create 调用 LLM

关键设计：

  • LangGraph 与「手写 asyncio 循环」双实现共存。LangGraph 的 astream() 会批量缓存消息，无法满足实时 SSE，因此默认走 _run_simplified() 路径
  • Critic 的智能路由通过修改 state["phase"] 触发：写为 RE_RESEARCHING 即重启 Scout，写为 REVISING 即仅触发 Writer
  • 每个 Agent 用各自的模型（见 config/llm_config.py）—— Scout 用 qwen-plus 求快，Writer 用 deepseek-v3.2 求质，互不干扰

2. ReAct 控制器与工具调用

入口：backend/app/service/react_controller.py

实现 4 阶段循环：Plan → Execute (并行) → Reflect → Synthesize。核心数据结构：

  • Tool：统一工具描述（name / description / parameters / handler）
  • SubQuery：LLM 生成的子查询，含 query / purpose / tool / priority
  • ReActContext：累积上下文（steps / observations / collected_data / insights / charts）

8 个内置工具：web_search / knowledge_search / text2sql / data_analyzer / chart_generator / stock_query / bidding_search / finish

并行执行通过 asyncio.gather + 任务异常隔离，单个工具失败不影响其他工具继续推进。

3. 双通道混合检索

组件 职责
web_search_service.py 封装博查 API，支持网页正文抽取
milvus_service.py Milvus 连接管理、collection 创建（COSINE + IVF_FLAT 索引）、向量增删查
embedding_service.py 调用 DashScope text-embedding-v4 生成 1024 维向量
retrieval_service.py 上层统一入口，融合两通道结果，调用 DashScope Rerank 重排
docmind_service.py 阿里云 DocMind 文档智能解析
document_service.py 文档分片 (jieba 中文 / nltk 英文)、入库流水线

4. Text2SQL 引擎

入口：backend/app/service/text2sql_service.py

特点：

  • Schema-aware：将 industry_stats / company_data / policy_data 三张表完整描述（含示例数据）注入 prompt，显著降低生成失败率
  • visualization_hint 自动推荐：时间序列 → line，分类比较 → bar，占比 → pie，详情 → table
  • 无数据库连接时优雅降级到 mock 数据，开发与演示场景都能跑

5. SSE 流式协议与检查点恢复

入口：backend/app/router/research_router.py + service/checkpoint_service.py

SSE 事件协议（前端可识别的 type 列表）：

type 时机 关键字段
research_start 研究开始 query / session_id / search_web / search_local
research_resumed 从检查点恢复 phase / session_id
phase 阶段切换 phase / content
thought Agent 推理 step / content / confidence
action 工具调用 step / tool / params
observation 工具结果 step / tool / success / result
search_result_item 单条搜索结果 result
chart 图表生成 id / title / echarts_option / image_base64
checkpoint_saved 检查点写入 phase / session_id
research_cancelled 用户取消 message
research_complete 研究结束 final_report / quality_score / facts_count / charts_count / iterations / references
error 异常 content

检查点表结构：research_checkpoints (id, session_id, user_id, state_json, ui_state_json, final_report, status, created_at)，前端通过 ui_state_json 直接重建研究面板。

6. 行业数据采集层

入口：backend/app/service/scheduler_service.py + 定时采集服务

  • APScheduler 启动定时任务：每天定时跑 news_collection_service 与 bidding_service
  • 关键词来源 config/industry_config.py：4 个预设行业，每个行业有独立的 news_keywords / bidding_keywords / research_keywords
  • 去重策略：按 (URL / 标题哈希) 入库时去重，避免相同新闻多次写入
  • 入库后立即向量化，落入对应行业的 Milvus collection，新闻同时进入 PostgreSQL 用于按时间/关键词检索

7. 长期记忆系统

入口：backend/app/service/memory_service.py + models/user.py::LongTermMemory

  • 记忆维度：preference (偏好) / focus_entity (关注实体) / research_topic (研究主题) / fact (用户陈述事实)
  • 写入时机：每轮对话结束后，由 LLM 提取「值得记忆的内容」并归类
  • 召回时机：新会话第一轮，按 user_id 召回 Top-K 长期记忆 + embedding 相似度排序，注入 system prompt

8. 招投标 / 股票 / 政策外部数据

服务 路由 数据来源
bidding_service.py /bidding/* 阿里云市场招投标 API，按地区/类别/关键词搜索
stock_service.py 通过 ReAct 工具调用 聚合数据股票 API，支持代码与名称模糊匹配
policy_search_service.py /research/* 内部调用 自研政策搜索，命中 policy_data 表后再做外部补充搜索

📊 评测体系

完整评测设计文档见 eval/eval.md。摘要如下：

数据集

Benchmark 题量 用途
IndustryBench (自建) 35 题 (11 领域 × 3 难度 + 长尾) 行业场景能力测试
HotpotQA-DR (改造版) 50 题 跨文档推理泛化性

每题 schema 含 expected_aspects（结构覆盖）+ gold_facts（事实准确）+ forbidden_claims（幻觉诱捕）三类标注。

三层指标

6 项消融实验

# 变量 关键问题
1 Critic 开/关 对抗审核对幻觉率影响
2 Web only / KB only / Web+KB 检索通道贡献
3 max_iterations = 1/2/3 迭代收敛点
4 Qwen-Plus / Qwen-Max / DeepSeek-V3 模型规模与成本权衡
5 IndustryBench vs HotpotQA-DR 跨 benchmark 泛化性
6 全量基线 论文级总表

设计哲学

  1. 可复现 — 同 config 跑两次结果一致，trace 全量落 SQLite
  2. 可对比 — 所有实验写入同一 trace store，跨配置 diff
  3. 可诊断 — 不止给总分，能下钻到「哪一题、哪个 Agent、失败原因」
  4. 可证伪 — 带 CI + Cohen’s d + Wilcoxon，避免「跑一次分高 0.3 就吹提升」
  5. 可外挂 — 评测层独立于业务代码，仅通过 service.research_sync() 接触业务

Agent记忆系统深度解析
Agent记忆系统深度解析在研究Agent架构的过程中，记忆系统是最让我着迷的部分。它远比”存储对话历史”这个简单概念复杂得多，本质上是一个精密的上下文管理系统。这篇文章记录了我对Agent记忆系统的深度理解。 记忆系统在Agent架构中的定位Agent通常由四大核心模块构成：LLM（大语言模型）、工具层、记忆系统、推理和执行系统。其中，记忆系统扮演着”大脑海马体”的角色——它不负责思考，但负责为思考提供必要的上下文支撑。 理解记忆系统的关键在于抓住一个本质：记忆系统的本质是上下文管理系统。它的核心任务是：在控制上下文长度的同时，保留关键信息，让Agent在任意时刻都能获得做出正确决策所需的背景知识。 这个视角很重要。如果我们把记忆系统仅仅理解为”存储聊天记录”，就会错过很多设计上的精妙之处。 短期记忆：不是”聊天”那么简单短期记忆的概念容易被误解。很多人会把它等同于”当前对话”，但这种理解过于粗糙。更准确的划分是三层结构： 即时工作记忆：这是Agent当前正在处理的信息，类似于人类工作记忆中的”焦点”。比如正在处理的一个复杂任务拆解、刚刚调用的工具返回结果、需要推理的中间步骤等...

HelloAgents 框架记忆系统设计解析 摘要：记忆是智能体的核心能力之一。HelloAgents 框架借鉴认知科学的记忆模型，设计了四层记忆架构，并整合 RAG 系统实现知识检索。本文深入解析这一设计的技术实现与工程考量。 为何智能体需要记忆大语言模型存在两个根本性局限：无状态导致的对话遗忘和内置知识的局限性。 对话遗忘问题LLM 设计上是无状态的，每一次请求都是独立的计算。这意味着： 上下文丢失：长对话中早期信息可能因窗口限制而丢失 个性化缺失：无法记住用户偏好和习惯 学习能力受限：无法从过往经验中学习 一致性问题：多轮对话可能出现前后矛盾 知识局限问题LLM 的知识完全来自训练数据，存在： 知识时效性：训练数据有时间截止点 专业领域深度：通用模型在特定领域知识不足 事实准确性：可能产生幻觉 可解释性：缺乏信息来源追溯 从认知科学获得启发人类记忆系统是多层级的认知结构，这为设计智能体记忆提供了理论框架。 人类记忆的层次结构 记忆类型 持续时间 容量 功能 感觉记忆 0.5-3 秒 巨大 暂存感官信息 工作记忆 15-30 秒 7±2 项目 当前任务...
