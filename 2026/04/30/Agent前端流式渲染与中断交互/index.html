Agent前端流式渲染与中断交互

用户对 Agent 的“聪明”感知，很多时候不来自答案质量，而来自等待过程是否可控。

先说结论

流式渲染不是锦上添花，它是 Agent 交互的主路径。

如果前端只做一次性整包返回，长任务会被用户感知成“卡死”。

中断能力也不只是“停止按钮”，而是系统对用户控制权的承诺。

先把基础知识补齐

常见流式协议有 SSE、WebSocket、HTTP chunked。

Agent 场景更看重三个能力：

  • 增量输出：token 级或段落级推送。
  • 状态事件：思考中、调用工具、等待确认、已完成。
  • 中断信号：用户能在任意时刻请求取消。

没有状态事件，流式只会变成“打字机动画”。

这个问题到底在讨论什么

我们不是在讨论 UI 特效。

我们讨论的是“前端如何把不确定、可回滚、可暂停的后端执行过程正确呈现”。

Agent 输出不是静态文档，而是持续演化状态机。

从基础往外延伸

可维护实现通常分三层：

  1. 传输层：负责事件可靠送达与重连。
  2. 协议层：定义事件类型、顺序号、幂等键。
  3. 渲染层：把事件映射成可理解的 UI 状态。

关键点在协议层。

如果没有顺序号和幂等键，断线重连后很容易重复渲染或丢消息。

这个问题为什么容易想错

误区一是“只流文本，不流状态”。

用户看到文字在滚动，不代表知道系统正在做什么。

误区二是“点击停止就前端停显示”。

真正的停止必须向后端传播取消信号，并终止工具调用链。

误区三是“重连后从头再来”。

长任务从头再来会放大成本，也破坏用户信任。

回到第一性原理

交互系统的底层约束是人类注意力和等待耐受。

当用户无法判断系统状态时，会默认系统已失败。

所以前端必须持续回答三个问题：

  • 现在做到哪一步了。
  • 为什么还没结束。
  • 我还能不能干预。

真正的机制是什么

高质量前端通常包含：

  • 事件日志面板：可折叠查看关键步骤与工具调用。
  • 可恢复流：断线后按最后序号续传。
  • 可取消任务：取消请求具备全链路传播。
  • 最终一致落盘：流式中间态与最终答案一致归档。

我现在怎么判断

检查一个 Agent 前端是否成熟，我看这五项：

  1. 是否有标准化事件协议而非纯文本流。
  2. 是否支持断线续传而非全量重跑。
  3. 取消后端是否真实停止而不是假停止。
  4. 中间态是否可追溯到最终结果。
  5. 错误是否区分网络问题、模型问题、工具问题。

总结

流式渲染解决的是可见性，中断交互解决的是控制权。

两者一起，才是用户对 Agent “可用”而非“能演示”的分水岭。

深度思考

前端流式体验的本质不是“看起来更快”，而是让用户持续拥有控制感。人在等待时最怕的不是慢，而是不确定：不知道系统是否卡死，不知道还能不能干预。很多实现只把文本按 token 打出来，却没有暴露任务状态，用户依旧焦虑。深层问题在于把流式当渲染问题，而不是协议问题。真正可用的流式系统必须有状态事件、顺序号、断线续传和取消语义，确保前端展示与后端执行一致。中断也不能只是 UI 停止按钮，必须能传递到工具调用链并真正终止副作用。更进一步，流式日志本身就是可观测资产：它记录了任务演化过程，可用于复盘和评测。做好这层后，你得到的不只是更顺滑的界面，而是可解释、可控、可回放的交互基础设施。

最后给一个可执行标尺：把这套方法交给一个不在场的同事，他能否仅凭日志、配置和文档，在一小时内复盘决策并复现实验结果。如果做不到，说明系统仍依赖个人经验而不是工程能力。继续把隐性判断显式化，直到可测、可审计、可回滚。
