深入浅出 MCP

MCP 最值得讲的，不是“模型终于能调工具了”，而是外部世界第一次被整理成了一套对 Agent 友好的标准接口。

先别急着把 MCP 讲成“AI 的 USB-C”

“MCP 像 AI 世界里的 USB-C”这句话传播得很广，因为它好记，也不算错。问题是，它会把一件本来很具体的工程问题，说成一个太轻的类比。

对 Claude Code 这种 Agent 来说，真正的难点从来不是“能不能发一个函数调用”，而是三件更麻烦的事：

  • 它怎么知道外部系统到底提供了什么能力。
  • 它怎么区分哪些东西是上下文，哪些东西是动作。
  • 它怎么在权限、认证、上下文预算这些硬约束里工作。

如果没有一套统一约定，每接一个系统，都得手写一套接法。文件系统一套，GitHub 一套，数据库一套，监控平台又一套。每个系统都要各自定义认证方式、参数格式、结果结构、错误处理、权限边界。模型层看起来像“工具变多了”，宿主层其实是在持续堆适配器。

MCP 想解决的，就是这层适配器地狱。

所以这篇文章不从比喻开始，而是从基础开始。先把协议本身讲清楚，再回到 Claude Code，看它为什么一接上 MCP，就不再只是一个“会聊天的代码助手”。

先把地板铺平：MCP 到底在定义什么

没有协议的时候，Agent 接外部世界靠的是手搓适配

先看最朴素的情况。一个 Agent 想访问外部世界，至少要处理四类问题：

  1. 连接问题：本地进程怎么起，远程服务怎么连。
  2. 描述问题：外部能力叫什么，接收什么参数，返回什么结构。
  3. 边界问题：什么动作需要授权，什么结果可以直接喂回模型。
  4. 生命周期问题：连接断了怎么办，能力变了怎么办，会话怎么恢复。

没有标准协议时，这四类问题都得一套一套重做。你可以把这种模式理解成“每接一个系统，就重新发明一遍小型集成框架”。

MCP 的价值，不是发明了“工具调用”这件事，而是把这些公共问题抽出来，变成了统一的协议层。

在 MCP 里，Claude Code 首先是宿主，不只是一个聊天窗口

官方文档把参与者拆成三层：host、client、server。

  • host 是宿主应用，比如 Claude Code、Claude Desktop 这类真正承载用户会话的程序。
  • client 是宿主里专门负责和某个 MCP server 通信的那一层。
  • server 是暴露能力的一端，可能是本地进程，也可能是远程服务。

这层拆分很关键。因为模型本身并不直接“连外部世界”。真正负责建立连接、保存能力清单、做权限判断、回收资源的，是 Claude Code 这个宿主。

换句话说，Claude Code 不是简单地把 MCP 当成一个插件市场。它更像一个运行时控制面：每接一个 MCP server，Claude Code 都要替模型管理一条受约束的工作链。

真正交换的东西分两层：数据层和传输层

MCP 官方架构把协议分成两层：

  • 数据层：定义消息语义，基于 JSON-RPC 2.0，负责 lifecycle、primitives、notifications 这些“到底在说什么”的问题。
  • 传输层：定义消息怎么送达，负责 stdio、Streamable HTTP、认证、连接建立这些“到底怎么送”的问题。

这个拆法特别像网络工程里的分层思想。它的好处是，能力描述可以稳定下来，底下走本地进程还是远程流式连接，可以分别处理。

在 Claude Code 里，这个分层很有现实意义：

  • 用 stdio 时，宿主拉起本地子进程，通过标准输入输出通信，适合本机工具。
  • 用 Streamable HTTP 时，宿主要处理远程连接、认证头、超时、重连、服务端流式响应，适合在线服务。

MCP 规范还专门提醒，HTTP 传输下服务端应该校验 Origin，本地部署时最好只绑 localhost，否则会暴露 DNS rebinding 风险。这说明传输层不是一个实现细节，而是安全边界的一部分。

一条 MCP 连接不是“直接开调”，而是先握手

MCP 是有状态协议。也就是说，连接建立后不是立刻开始干活，而是要先做 lifecycle 管理。

这一步至少要确认三件事：

  1. 双方说的是哪一版协议。
  2. 双方各自支持哪些能力。
  3. 这条连接后面允不允许动态更新、用户输入、资源订阅之类的扩展行为。

官方规范把初始化放在最前面，不是为了仪式感，而是因为后面很多行为都依赖这次协商。比如服务端如果支持 tools/list_changed 通知，就得在能力声明里先说清楚；客户端如果支持 sampling 或 elicitation，也得先告诉对方。

因此，MCP 不是“给我一个工具名，我现在就调”。它先建立一段双方都认账的会话，再在这段会话里暴露能力。

Tools、Resources、Prompts，其实是在分三种完全不同的需求

很多人一提 MCP，脑子里只剩 tools。这不够。

MCP 在服务端侧定义了三类核心原语：

  • Tools：可执行动作。比如读数据库、调 API、改状态。
  • Resources：可引用上下文。比如文件内容、文档、schema、历史记录。
  • Prompts：可复用的交互模板。比如某类任务该怎么组织输入，怎么带少量示例。

这三层的分工非常重要。

Tool 解决的是“帮我做事”。
Resource 解决的是“先给我看东西”。
Prompt 解决的是“这一类事情最好怎么和模型配合”。

在 Claude Code 里，这三层会落成完全不同的使用体验：

  • tools/list 发现出来的能力，会变成模型可调用的动作集合。
  • resources 可以通过 @server:protocol://resource/path 这种方式被引用进上下文。
  • prompts 会以 /mcp__servername__promptname 的形式出现在斜杠命令里。

如果不把这三层分开，你最后很容易把一切都塞进 tool calling，结果是上下文和动作混在一起，既难控，也难用。

Sampling、Elicitation、Notifications 让它不是单向工具箱

MCP 还有一层常被忽略的设计：它不是“服务端只会被调用”的单向模型。

客户端侧也可以暴露能力给服务端，比如：

  • sampling：服务端可以请求宿主代发一次模型生成。
  • elicitation：服务端可以在中途向用户要结构化输入。
  • logging：服务端可以把日志发回宿主。

此外，协议支持通知机制。比如服务端能力变了，可以主动发 list_changed，客户端收到后再刷新自己的能力视图。

这件事很重要，因为它意味着 MCP 管理的不是“函数目录”，而是一条活着的工作链。这个链路里，服务端、宿主、用户都可能在不同时间点重新介入。

把术语拿掉，MCP 的基本原理其实很朴素

把各种名词都扔掉，MCP 本质上在做五件事：

  • 定义输入：外部系统怎么把自己的能力和数据描述出来。
  • 定义状态：连接一旦建立，双方怎么知道彼此支持什么。
  • 定义动作：哪些能力是可执行的，参数长什么样，结果怎么回。
  • 定义反馈：能力变化了、需要用户确认了、需要补充输入了，该怎么通知。
  • 定义边界：传输、认证、权限、输出体量这些限制，宿主应该在哪里收口。

所以 MCP 不是在替模型“增加智力”，而是在替宿主“整理外部世界”。

这个原理一旦想明白，你就不会再把 MCP 看成一个花哨的连接器。它更像一层面向 Agent 的 I/O 协议：外部世界不再以“各种零散 API”的形式出现，而是以“可发现、可协商、可受控的能力面”出现。

回到 Claude Code：一条 MCP 调用在它内部怎么跑

Claude Code 是理解 MCP 的好例子，因为它把宿主责任做得很完整。

先看一个最基础的项目级配置：

这只是入口。真正跑起来时，大致会经过下面七步。

第一步：决定这条连接属于哪个 scope

Claude Code 把 MCP server 分成 local、project、user 三层。

  • project 级通常写进仓库根目录的 .mcp.json，适合团队共享。
  • user 级存在用户配置里，跨项目可见。
  • local 级只对当前项目、当前用户生效。

这一步解决的不是存储位置，而是治理范围。谁能看到它，谁要审批它，出了问题谁来承担边界，都是这里定的。

第二步：宿主建立传输，拉起本地进程或连接远程服务

如果是 stdio server，Claude Code 会像启动一个工具进程一样把它拉起来；如果是 HTTP server，就要处理 URL、headers、认证和连接超时。

这一步也是为什么 MCP 不是单纯的“工具声明文件”。真正需要操心连接健康和安全细节的是宿主，不是模型。

第三步：先做初始化协商，再谈后续能力

Claude Code 会先和 server 确认协议版本和能力声明。只有在这一步双方达成一致后，后续的 tools/list、resources/read、prompts/list 才有意义。

如果服务端不支持当前版本，它应该明确报错，而不是静默失败。这个设计看起来保守，实际上让宿主的行为更可调试。

第四步：按原语发现能力，而不是一股脑塞进上下文

协商完成后，Claude Code 会去发现服务端暴露的工具、资源、prompt。

这里最有意思的地方在于：它并不需要把所有能力都立刻完整放进当前会话。Claude Code 官方 SDK 文档明确提到，MCP tools 多起来后，工具定义本身会吃掉上下文窗口，因此默认启用了 tool search，把工具定义延后到真正需要时再加载。

这说明宿主在做的不只是“工具注册”，而是上下文预算管理。

第五步：把三类原语映射成三种不同的交互表面

在 Claude Code 里，MCP 原语最后会变成三种不同用法：

  • Tools 进入模型可调用动作集合。
  • Resources 通过 @ 引用进入上下文。
  • Prompts 以斜杠命令暴露出来。

第六步：真正执行前，先过权限和认证

Claude Code 的安全模型不是附加功能，而是宿主层的一部分。

官方安全文档强调，它默认采用严格权限模式；编辑文件、执行命令、使用某些外部能力前，宿主会要求明确许可。项目级 .mcp.json 里的 server 也需要审批。换句话说，MCP 给了能力入口，但不是能力白名单。

这也是为什么我一直说，MCP 的重点不只是“能接”，而是“谁来收边界”。在 Claude Code 里，边界是宿主来收，不是交给模型自己判断。

第七步：结果回流到会话，再决定下一步

tools/call 返回的不是一个随便拼的字符串，而是结构化内容数组。Claude Code 拿到结果后，会决定下一步是继续调工具、补拿资源、请求用户输入，还是直接回答。

到这里，MCP 才真正完成闭环：连接、协商、发现、执行、回流。

一个最容易被问到的问题：为什么 OpenClaw 有时更愿意走 CLI，而不是把一切都塞进 MCP

这是个特别好的问题，因为它能逼我们区分两件经常被混在一起的事：

  • “把外部能力暴露给宿主使用”
  • “把一个现成的 Agent runtime 接进来，让它自己跑长会话”

OpenClaw 的官方文档其实把这两件事分得很清楚。

OpenClaw 并没有放弃 MCP，它只是没有把 MCP 用到不该用的地方

OpenClaw 的 VISION.md 明确写了两点：

  • 它支持 MCP，既可以把自己暴露成 MCP server，也可以把 MCP 当作运行时集成面。
  • 它追求的是“务实的 MCP 支持”，不去重复已有的 agent、tool、ACPX、plugin 路径。

这句话很重要。它说明 OpenClaw 的态度不是“不要 MCP”，而是“别让 MCP 去承担别的层次早就更适合承担的工作”。

官方 openclaw mcp 文档也说得很直接：
openclaw mcp serve 适合让 Codex、Claude Code 这类 MCP client 去访问 OpenClaw 侧的会话、消息历史、审批工具；但如果是 OpenClaw 自己来托管一个 coding harness session，就该用 openclaw acp。

也就是说，在 OpenClaw 眼里：

  • MCP 更适合暴露能力面。
  • ACP/CLI runtime 更适合托管一个正在运行的代理会话。

原因一：完整的编码代理，不只是“几个工具”

把一个现成的编码代理，比如 Claude Code、Codex、Gemini CLI，完整接进系统时，你接的不是一个 tool set，而是一整套运行时行为：

  • 会话创建和恢复
  • 长流式输出
  • 中断和取消
  • 原生权限模型
  • 自己的 slash commands
  • 自己的模型选择和认证逻辑

这类东西如果硬压成 MCP tools，会有明显语义损耗。因为 MCP 很擅长描述“宿主向外部能力发请求”，不那么适合无损承载“另一个完整代理 runtime 的全部会话语义”。

OpenClaw 的 ACP 文档本身就侧面说明了这个问题。它在兼容矩阵里列得很清楚：initialize、newSession、prompt、cancel 这些是桥接重点；而更深的会话模式、历史重建、终端方法、per-session MCP servers 等能力，要么是 partial，要么是 unsupported。

这恰好说明，托管一个编码 runtime，本质上是在托管会话，不是在暴露几个离散工具。

原因二：CLI 能直接复用现成产品的认证和订阅路径

OpenClaw 的 model providers 文档给了一个很现实的信号：当你想要本地 CLI backend 时，可以把运行时策略设成 claude-cli、google-gemini-cli 这类本地 CLI runtime。

同一份文档里还有一句更关键的话：OpenClaw 现在偏向复用 Claude CLI 和 claude -p，在可用时优先走这条路。

这背后不是协议优雅性，而是产品现实：

  • CLI 已经有官方登录流。
  • CLI 已经绑定了厂商认可的认证方式。
  • CLI 已经把一些模型选择、权限和工具行为打包好了。

如果宿主能直接复用这条成熟链路，就没有必要重新把同一套东西再包成一层 MCP wrapper。

原因三：CLI runtime 保留了原生行为，集成成本更低

当你复用现成 CLI 时，很多行为是“原封不动借过来”的：

  • 它原生怎么流式输出，集成层就怎么接。
  • 它原生怎么登录，集成层就怎么复用。
  • 它原生怎么约束工具，集成层通常不必重新建一遍语义。

这比把一个完整 agent 再拆成 MCP server，再让宿主重新理解一遍，往往更省。

换句话说，CLI runtime 胜出的地方，不是比 MCP 更“先进”，而是它在接入一个现成产品时，语义损耗更少。

原因四：MCP 适合能力互操作，CLI 适合产品复用

这是最核心的分界线。

如果你想做的是：

  • 让 Claude Code 访问外部文档
  • 让 OpenClaw 暴露消息会话给别的宿主
  • 让一个 Agent 统一接 GitHub、数据库、搜索、监控这些能力

那 MCP 很合适，因为你需要的是统一能力面。

如果你想做的是：

  • 把 Claude Code 整个作为一个现成 runtime 托管进来
  • 复用 Codex 或 Gemini CLI 的原生能力
  • 保留供应商产品已经做好的认证和会话行为

那 CLI 或 ACP 往往更顺手，因为你需要的是完整 runtime，不是能力目录。

那 CLI 会不会把 MCP 取代掉

我的判断是：不会。但它会吃掉一部分原本不该硬做成 MCP 的集成。

原因很简单，二者解决的问题层次不同。

CLI 擅长借用一个现成产品

CLI 是产品表面。它的优点是：

  • 直接复用现成认证和订阅
  • 尽量保留原生运行时语义
  • 对特定产品的深度接入成本低

所以在“把 Claude Code / Codex / Gemini 这类东西直接接进另一个系统”这条线上，CLI 或 ACP 这种路径确实很强。

MCP 擅长做跨产品的公共接口

MCP 是协议表面。它的优点是：

  • 有统一的能力发现
  • 有统一的 tools / resources / prompts 分层
  • 有统一的传输和能力协商模型
  • 宿主可以把不同来源的能力收到一个公共控制面里

CLI 做不到这一点。CLI 天然是产品私有接口，不会自动变成通用互操作层。

最能说明问题的一点是：Claude Code 自己也能当 MCP server

Claude Code 官方文档明确支持：

也就是说，CLI 产品本身并没有把 MCP 干掉，反而愿意把自己再暴露成一个 MCP surface，供别的宿主接入。

这恰好说明两者不是替代关系，而是分层关系：

  • CLI 是一个具体产品的原生入口。
  • MCP 是不同产品之间更稳定的公共接口。

在接完整 runtime 时，CLI 可能比 MCP 更省。
在做跨工具生态互操作时，MCP 仍然更像基础设施。

真到工程里，什么时候选 MCP，什么时候直接接 CLI

可以用一张很实用的表来判断：

场景 更适合的路径 原因
让宿主访问外部系统的能力、文档、数据 MCP 统一发现、统一权限、统一上下文入口
把一个现成 coding agent 当作完整 runtime 接进来 CLI / ACP 要保留完整会话语义、认证和原生行为
让多个不同工具在同一个宿主里共存 MCP 宿主能用一套控制面治理它们
借用单一厂商产品已有的登录与订阅 CLI 少一层重打包，产品语义损耗更小
需要跨产品、跨团队共享一套连接规范 MCP 比私有 CLI 更适合长期互操作

一句话说：

要“借一个完整代理”，先想 CLI。
要“接一片可组合能力”，先想 MCP。

我现在怎么看 MCP

写到这里，我对 MCP 的判断已经很稳定了。

它不是某种让 Agent 突然变聪明的咒语。它做的事情更基础，也更值钱：把外部世界整理成一套宿主可以治理的能力面。

所以我现在看一个集成方案，先不问“连了多少服务”，而先问四个问题：

  1. 它是在暴露能力，还是在托管完整 runtime。
  2. 它有没有把动作、上下文、交互模板分开。
  3. 宿主有没有真正收住权限、认证、输出体量和上下文预算。
  4. 它选 MCP 还是 CLI，是因为问题本身适合，还是因为团队只会其中一种。

Claude Code 适合拿来讲 MCP，不是因为它功能最多，而是因为它把宿主该负的责任做得很清楚。
OpenClaw 适合拿来讲边界，不是因为它反对 MCP，而是因为它很诚实地承认：有些事情该交给 MCP，有些事情更该交给 CLI 或 ACP。

这恰好也是我对这个问题最后的结论。

MCP 不会被 CLI 取代。
但凡是“完整 runtime 托管”这种问题，未来很多团队会越来越少地硬包一层 MCP，而是直接复用现成 CLI。
协议层和产品层会一起存在，各自吃掉自己最擅长的那部分复杂度。

    2. 1.2. 先把地板铺平：MCP 到底在定义什么
      1. 1.2.1. 没有协议的时候，Agent 接外部世界靠的是手搓适配
      2. 1.2.2. 在 MCP 里，Claude Code 首先是宿主，不只是一个聊天窗口
      3. 1.2.3. 真正交换的东西分两层：数据层和传输层
      4. 1.2.4. 一条 MCP 连接不是“直接开调”，而是先握手
      5. 1.2.5. Tools、Resources、Prompts，其实是在分三种完全不同的需求
      6. 1.2.6. Sampling、Elicitation、Notifications 让它不是单向工具箱
    3. 1.3. 把术语拿掉，MCP 的基本原理其实很朴素
    4. 1.4. 回到 Claude Code：一条 MCP 调用在它内部怎么跑
      1. 1.4.1. 第一步：决定这条连接属于哪个 scope
      2. 1.4.2. 第二步：宿主建立传输，拉起本地进程或连接远程服务
      3. 1.4.3. 第三步：先做初始化协商，再谈后续能力
      4. 1.4.4. 第四步：按原语发现能力，而不是一股脑塞进上下文
      5. 1.4.5. 第五步：把三类原语映射成三种不同的交互表面
      6. 1.4.6. 第六步：真正执行前，先过权限和认证
      7. 1.4.7. 第七步：结果回流到会话，再决定下一步
    5. 1.5. 一个最容易被问到的问题：为什么 OpenClaw 有时更愿意走 CLI，而不是把一切都塞进 MCP
      1. 1.5.1. OpenClaw 并没有放弃 MCP，它只是没有把 MCP 用到不该用的地方
      2. 1.5.2. 原因一：完整的编码代理，不只是“几个工具”
      3. 1.5.3. 原因二：CLI 能直接复用现成产品的认证和订阅路径
      4. 1.5.4. 原因三：CLI runtime 保留了原生行为，集成成本更低
      5. 1.5.5. 原因四：MCP 适合能力互操作，CLI 适合产品复用
    6. 1.6. 那 CLI 会不会把 MCP 取代掉
      1. 1.6.1. CLI 擅长借用一个现成产品
      2. 1.6.2. MCP 擅长做跨产品的公共接口
      3. 1.6.3. 最能说明问题的一点是：Claude Code 自己也能当 MCP server
