Agent任务队列与幂等执行

Agent 一进生产，最难的不是“能跑起来”，而是“重复执行时不出脏数据”。

先说结论

任务队列解决吞吐，幂等设计解决正确性。

没有幂等，重试机制会把偶发失败放大成业务事故。

先把基础知识补齐

队列系统常见能力是削峰、异步、重试、延迟执行。

Agent 任务天然长链路、多步骤，失败点很多。

所以你必须默认“至少一次执行”会发生，而不是奢望“恰好一次”。

这个问题到底在讨论什么

不是 Redis 还是 Kafka 的选型问题。

核心是同一任务被重复消费时，系统是否还能保持一致结果。

从基础往外延伸

稳定方案通常有三把锁：

  • 业务幂等键：由用户意图或任务指纹生成。
  • 步骤幂等键：每个子步骤单独可去重。
  • 外部调用幂等键：调用支付、工单、邮件等外部系统时透传。

这个问题为什么容易想错

误区是把“消息去重”当成“业务幂等”。

消息去重只能减少重复消费，不保证副作用不重复。

回到第一性原理

幂等的本质是把“操作”变成“状态覆盖”。

追加式副作用最危险，覆盖式状态最稳。

真正的机制是什么

推荐闭环：

  1. 入队前生成全局幂等键。
  2. 消费前查幂等表，命中则短路返回。
  3. 执行步骤时记录状态机迁移。
  4. 成功后落最终状态并写审计日志。

我现在怎么判断

  1. 同一请求重放 10 次结果是否一致。
  2. 任务中途失败重试是否重复副作用。
  3. 人工补偿后是否还能安全重放。
  4. 幂等键是否跨服务传递。
  5. 审计日志是否能还原每次状态迁移。

总结

队列让系统更快，幂等让系统不烂。

Agent 后端稳定性的分水岭，就在这两者是否一起设计。

深度思考

幂等设计最容易被低估，因为系统在小流量时看起来“偶尔重试也没事”。但当并发和失败叠加，重复执行会迅速把副作用放大成业务事故。深层原因是很多架构把队列可靠投递误当成业务可靠完成，实际上消息层去重无法替代业务层幂等。真正稳的系统要把幂等键嵌入业务语义，而不是仅依赖消息 ID：同一用户意图、同一步骤、同一外部调用都应有可追踪的去重边界。再往下看，幂等不是技术细节，而是时间维度的一致性承诺。你要回答的是“同一件事在不同时间被重复触发，结果是否仍然唯一”。当你把状态迁移显式化并记录每次副作用，重试就从风险来源变成恢复能力。队列负责把任务送达，幂等负责把结果守住。

最后给一个可执行标尺：把这套方法交给一个不在场的同事，他能否仅凭日志、配置和文档，在一小时内复盘决策并复现实验结果。如果做不到，说明系统仍依赖个人经验而不是工程能力。继续把隐性判断显式化，直到可测、可审计、可回滚。
