Agent长任务编排与补偿事务

Agent 一旦跨系统执行长任务，失败不是例外，而是常态。

先说结论

长任务可靠性不是靠“少失败”，而是靠“失败可恢复”。

补偿事务是 Agent 编排系统的必选项，不是可选增强。

先把基础知识补齐

长任务通常跨多个步骤：规划、检索、调用、写入、通知。

任何一步失败都可能让整体进入半完成状态。

半完成如果不可见、不可补偿，就会留下业务脏状态。

这个问题到底在讨论什么

不是工作流引擎选型问题。

核心是“跨步骤副作用是否可逆，或至少可补偿”。

从基础往外延伸

推荐把每个步骤声明为：

  • 前置条件
  • 执行动作
  • 成功标记
  • 失败补偿动作

补偿动作不一定是反向操作，也可以是对账、冲销、人工工单。

这个问题为什么容易想错

误区一：只做重试，不做补偿。

重试能修瞬时失败，修不了逻辑失败和外部副作用失败。

误区二：补偿逻辑写在人工 SOP，不在系统里。

这会导致恢复慢且不可追踪。

回到第一性原理

分布式系统不存在免费的一致性。

你要么在前面花成本设计补偿，要么在后面付更大事故成本。

真正的机制是什么

成熟做法：

  1. 任务状态机显式化。
  2. 每步产出持久化检查点。
  3. 失败触发自动补偿或人工接管分支。
  4. 补偿完成后可安全重入。

我现在怎么判断

  1. 任一步骤失败后是否能给出明确恢复路径。
  2. 任务是否支持从检查点继续。
  3. 补偿动作是否有成功判定标准。
  4. 人工接管点是否清晰。
  5. 恢复后是否可再次自动执行。

总结

长任务编排的核心不是流程图漂亮，而是失败后不留烂账。

把补偿事务内建，系统才称得上生产级。
