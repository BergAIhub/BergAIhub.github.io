Agent目标函数与反馈回路笔记

很多 Agent 不是能力不够，而是目标函数写得含糊，导致它每一步都看起来合理，整体却持续跑偏。

先说结论

Agent 的核心不是“会不会调用工具”，而是“到底在优化什么”。

只要目标函数模糊，Agent 就会把局部最优当成全局最优。

真正稳定的 Agent，需要把反馈回路设计成可观测、可比较、可终止。

先把基础知识补齐

Agent 和普通脚本最大的区别，是它会在不确定环境里持续做决策。

每一步决策都在回答同一个问题：下一步做什么，最可能让目标更接近完成。

所以它天然需要三个要素：

  • 目标状态：什么叫完成，什么叫更接近完成。
  • 行动集合：它可以做哪些动作，比如思考、调用工具、写文件、请求用户确认。
  • 反馈信号：动作之后，如何判断是进步还是退步。

这三个要素缺任何一个，Agent 都会退化成“看起来很忙”。

这个问题到底在讨论什么

我们讨论的不是“Prompt 怎么写更聪明”。

我们讨论的是一个更底层的问题：Agent 的行为是否被一个一致的优化目标约束。

如果没有这个约束，模型会自然倾向于输出“短期看起来最像正确答案”的内容。

这就是很多系统里常见的幻觉、过度自信、反复试错不收敛。

从基础往外延伸

一个实用的目标函数通常不是单一指标，而是加权组合。

例如在编程 Agent 里，常见目标是：

  • 任务完成度：是否满足用户请求。
  • 正确性：测试是否通过，行为是否符合预期。
  • 成本：时间、Token、工具调用次数。
  • 风险：是否触发安全边界，是否需要人工确认。

Agent 每一步都在做权衡。

如果你只强调“尽快完成”，它会跳过验证。

如果你只强调“绝对正确”，它会无限思考，不愿提交结果。

这个问题为什么容易想错

常见误区是把“更强模型”当成解决方案。

模型更强只能提升单步推理质量，不能自动补上系统层面的目标对齐。

另一个误区是把“多写规则”当成目标函数。

规则越多不代表目标越清晰。

规则之间如果冲突，Agent 只会在冲突处随机选边，表现成不稳定。

回到第一性原理

从第一性看，任何 Agent 系统都受三个硬约束：

  • 信息不完备：它永远不可能一次看到全部真实状态。
  • 反馈有延迟：很多错误要到后续步骤甚至上线后才暴露。
  • 计算资源有限：不可能无限探索所有路径。

因此目标函数设计的本质，不是追求“理论最优”，而是让有限资源下的行为尽量可控。

真正的机制是什么

稳定系统通常用一个循环：

  1. 根据当前状态估计最优下一步。
  2. 执行动作并采集反馈。
  3. 用反馈更新对任务状态的判断。
  4. 判断是继续、回滚、还是结束。

这个循环要成立，关键在于反馈信号质量。

反馈信号如果只来自“模型自评”，就会形成自我强化偏差。

反馈里必须混入外部证据，例如测试结果、命令输出、结构化校验、人工确认。

用这个模型重新看问题

你会发现很多“Agent 不稳定”的案例，其实不是推理错，而是反馈回路断裂。

比如：

  • 只看到了命令执行成功，没检查业务结果是否正确。
  • 写了代码但没跑测试，却认为任务完成。
  • 失败后不记录失败原因，下一轮重复同样错误。

这些都不是智商问题，是控制回路问题。

我现在怎么判断

评估一个 Agent 方案，我会先看五件事：

  1. 目标是否可度量，不是口号化描述。
  2. 每次行动后是否有外部可验证反馈。
  3. 是否存在明确终止条件，避免无限循环。
  4. 失败是否会被结构化记录并影响下一步决策。
  5. 在时间或成本受限时，是否有降级策略。

这五条都满足，系统通常不会差到哪去。

总结

Agent 的本质不是“会做很多事”，而是“在不确定里持续做正确权衡”。

权衡能否稳定，取决于目标函数和反馈回路是否闭合。
