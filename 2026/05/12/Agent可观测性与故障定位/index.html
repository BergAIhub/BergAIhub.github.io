Agent可观测性与故障定位

没有可观测性，Agent 出错时你只会看到“答案不对”，看不到“哪里先坏了”。

先说结论

Agent 监控不能只盯延迟和 QPS。

你需要同时观测模型行为、工具行为、业务行为三条链。

先把基础知识补齐

可观测性通常包含日志、指标、链路追踪。

在 Agent 场景里还要新增“决策事件”。

即：为什么选这个工具、为什么中断、为什么给这个结论。

这个问题到底在讨论什么

不是 ELK 还是 OpenTelemetry。

核心是一次失败能否在 5 分钟内定位到责任环节。

从基础往外延伸

建议统一 trace_id，贯穿：

  • 用户请求
  • Agent 计划步骤
  • 工具调用
  • 外部服务响应
  • 最终输出

没有统一 trace_id，多系统日志就无法拼接。

这个问题为什么容易想错

误区一：只采集错误日志，不采集关键决策点。

误区二：日志很全，但没有结构化字段，仍然难检索。

回到第一性原理

故障定位本质是缩小不确定性空间。

可观测性的价值不在“收集很多数据”，而在“快速排除错误路径”。

真正的机制是什么

高质量方案会做三件事：

  1. 关键事件结构化：step、tool、latency、status、error_class。
  2. 错误分层归因：模型错、工具错、配置错、数据错。
  3. 回放能力：能用同一输入重现关键流程。

我现在怎么判断

  1. 是否有跨服务统一 trace_id。
  2. 是否能按 error_class 出故障看板。
  3. 是否支持单次会话全链路回放。
  4. 是否能区分“答非所问”和“工具超时”。
  5. MTTR 是否在持续下降。

总结

可观测性不是运维附属品，它是 Agent 工程质量的主干。

没有它，就没有可验证的迭代速度。

深度思考

可观测性的深水区，不是“看见错误”，而是“解释错误为什么必然发生”。如果日志只记录结果，不记录决策上下文，你永远只能在事后猜测。Agent 场景里最昂贵的不是一次失败，而是同类失败反复发生却无法归因。很多团队会堆监控面板，但缺少统一事件语义，最后每个系统都说自己没问题。真正有效的可观测性要回答三件事：模型当时看到了什么、为什么选了这一步、外部世界给了什么反馈。只有这三条链路能对齐，故障定位才会从“搜日志”升级到“还原因果”。再往下看，可观测性的价值还在组织协作：当研发、算法、运维都看同一条事实链，争论会减少，迭代会加速。你不是在建设监控，而是在建设团队的共同现实。

最后给一个可执行标尺：把这套方法交给一个不在场的同事，他能否仅凭日志、配置和文档，在一小时内复盘决策并复现实验结果。如果做不到，说明系统仍依赖个人经验而不是工程能力。继续把隐性判断显式化，直到可测、可审计、可回滚。
