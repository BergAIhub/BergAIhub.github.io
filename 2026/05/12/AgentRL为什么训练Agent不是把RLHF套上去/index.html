﻿# AgentRL：为什么训练 Agent 不是把 RLHF 套上去

我这里说的 AgentRL，主要指 THUDM/Z.AI 开源的 AgentRL 框架，以及它背后的论文《AgentRL: Scaling Agentic Reinforcement Learning with a Multi-Turn, Multi-Task Framework》。它要解决的问题不是“怎么让模型回答更好听”，而是怎么让模型在多轮环境里用工具、查资料、写代码、验证结果，并且真的因为做对事情而被强化。

这个问题为什么突然变重要

过去我们谈大模型训练，经常围绕三件事：预训练、指令微调、RLHF。

预训练让模型学会语言和世界知识。指令微调让模型更像一个会听话的助手。RLHF 让模型输出更符合人类偏好。

但 Agent 出现以后，问题变了。

一个 Agent 不只是回答一句话。它可能要先读文件，再调用工具，再根据错误日志改代码，再运行测试，最后决定要不要回滚。这个过程有多轮交互，有环境状态，有工具失败，有中间选择，也有很长的延迟反馈。

这时候如果还只用“单轮问答”的训练方式，就会碰到一个很硬的限制：模型不是不会说答案，而是不知道怎么在环境里把事情做完。

AgentRL 的价值就在这里。它不是又一个聊天模型对齐技巧，而是把 RL 训练搬到 Agent 的真实工作流里。

先把几个基础概念拆开

1. RLHF 解决的是“回答像不像人喜欢”

RLHF 的经典目标是偏好对齐。

人类标注者或者奖励模型会告诉系统：两个回答里哪个更好。模型学到的是一种偏好分布，比如更有帮助、更安全、更礼貌、更完整。

这对聊天助手很重要，但它有一个隐含前提：任务通常在一次回答里结束。

用户问一个问题，模型给一个答案，然后奖励系统评价这个答案。这条链路比较短，状态也不复杂。

Agent 不是这样。

Agent 的失败经常不是最后一句话写得不好，而是中间某一步走错了。比如它查错文件、误解测试失败、调用工具参数错、看到报错后没有更新计划。最后答案可能很自信，但任务已经在第 3 步坏掉了。

所以训练 Agent，不能只盯着最终文本。要盯着行动过程。

2. RLVR 解决的是“结果能不能验证”

RLVR 通常指 Reinforcement Learning with Verifiable Rewards，也就是用可验证结果当奖励。

数学题算对了，代码测试过了，网页任务完成了，这些结果可以自动判断。它比纯偏好奖励更硬，因为奖励不是“我觉得这个回答更好”，而是“这个结果确实对”。

这很适合推理和代码任务。

但普通 RLVR 也有边界。很多训练还是围绕单任务、单环境、相对短的轨迹。比如给一道题，模型解出来，验证器给奖励。

奖励可能很晚才出现，而且中间每一步都可能影响后面。模型要学的不只是“答案是什么”，还包括“什么时候查资料、什么时候运行测试、什么时候停止、什么时候换方案”。

3. Agentic RL 训练的是“做事过程”

Agentic RL 可以粗略理解成：把 Agent 放进环境里，让它通过多轮行动拿到奖励。

这里的关键词是环境。

环境可以是代码仓库，可以是浏览器，可以是搜索系统，可以是数学验证器，也可以是一个需要多步操作的任务模拟器。模型不是一次性吐答案，而是在环境里做动作。

这会带来三个新问题。

第一，rollout 很慢。一次轨迹可能要调用很多工具，跑很多测试。训练吞吐会被环境拖慢。

第二，任务差异很大。代码、数学、检索、网页、工具调用的动作空间和奖励形式都不一样。

第三，奖励容易不稳定。环境失败、超时、工具抖动、验证器误判，都会让训练信号变脏。

AgentRL 这类框架要解决的，正是这些工程问题。

4. 多轮不是“多说几句”，而是状态在变

很多人听到 multi-turn，会以为只是对话轮数多一点。

对 Agent 来说，多轮的核心不是聊天更长，而是状态一直在变。

模型第一次看到的是初始环境。它执行一个动作以后，环境返回新信息。这个新信息会改变下一步选择。比如：

这不是普通上下文拼接。每一步都在改变未来可见的信息和可选动作。

所以 Agent 训练的关键，不是让模型背更多“最佳实践”，而是让它在状态变化里学会选择动作。

5. 多任务不是“数据集混一下”

AgentRL 论文强调 multi-task，不是简单把多个数据集混起来。

真正的问题是，不同任务的环境后端、动作格式、奖励函数、轨迹长度都不一样。如果每种任务都写一套训练逻辑，系统很快会碎掉。

一个可扩展的 Agent RL 框架，必须把“模型训练”和“环境执行”拆开。

训练侧关心策略更新、采样、优势估计、梯度。环境侧关心工具怎么调用、任务怎么验证、奖励怎么返回。

这两个部分解耦以后，才有可能把代码任务、网页任务、检索任务、推理任务放进同一个训练框架里。

6. 异步 rollout 是为了救吞吐

Agent 的环境交互很慢。

如果训练进程每次都等所有环境跑完，再统一更新模型，就会被最慢的任务卡住。一个任务卡在浏览器，一个任务卡在测试，一个任务超时，整批训练都被拖慢。

异步 rollout 的思路很直接：环境交互和模型更新不要死等彼此。

抛开框架，AgentRL 的第一性原理是什么

如果把名字都拿掉，AgentRL 做的是一件很简单的事：

给模型一个环境，让它观察状态，选择动作，拿到反馈，再根据反馈调整以后选择动作的概率。

输入是什么？

输入不是单个 prompt，而是当前环境状态：用户目标、历史动作、工具返回、文件内容、错误日志、验证结果。

输出是什么？

输出也不是单纯一段文本，而是下一步动作。这个动作可能是回答、搜索、打开文件、执行命令、修改代码、提交结果。

中间保存了什么？

保存的是轨迹。也就是模型在每个状态下做了什么动作，环境返回了什么，最后得到什么奖励。

受什么约束？

受环境速度、奖励质量、上下文长度、工具稳定性、任务分布、训练成本约束。

所以 AgentRL 的底层逻辑不是“让模型更聪明”。更准确地说，它是在训练模型形成一套行动策略：什么时候探索，什么时候验证，什么时候继续，什么时候停止。

这也是为什么它和普通 SFT 不一样。

SFT 学的是“看到这种输入，模仿这个输出”。AgentRL 学的是“在这种状态下，这个动作以后更可能把任务做成”。

AgentRL 的底层机制怎么跑起来

根据论文和开源仓库，AgentRL 的核心设计可以概括成三层：任务环境层、轨迹采样层、策略训练层。

环境层负责把任务变成可交互系统

环境层要回答几个问题：

模型能看到什么？

模型能做什么动作？

动作执行以后，环境返回什么观察？

什么时候结束？

奖励怎么计算？

比如代码任务里，动作可能包括读文件、改文件、运行测试。检索任务里，动作可能是搜索、打开网页、整理证据。数学任务里，动作可能主要是推理和提交答案。

轨迹采样层负责让模型多轮行动

一次 rollout 就是一条轨迹。

在 Agent 场景里，这条轨迹可能很长，而且每个任务长度不一样。

AgentRL 这类框架的难点，是怎么高效收集这些轨迹。同步采样会慢，完全异步又可能带来策略滞后。训练系统要在吞吐、稳定性、新鲜度之间折中。

这也是论文里强调 scaling 的原因。Agentic RL 不只是算法问题，它首先是系统问题。

策略训练层负责把反馈变成参数更新

有了轨迹以后，训练层要判断哪些动作值得强化。

如果最终任务成功，轨迹里的动作不一定每一步都好；如果最终失败，也不代表每一步都坏。这是长程 credit assignment 的老问题。

在 Agent 里它更明显。模型可能前面探索得很好，最后因为一个小错误失败；也可能一路瞎撞，最后侥幸通过。

所以训练时需要算法和工程一起兜住：限制策略更新幅度，过滤异常轨迹，处理超时和无效动作，稳定 reward 计算。

为什么它不是“把 PPO 套到 Agent 上”

PPO 或其他 RL 算法只是训练层的一部分。

AgentRL 真正麻烦的地方在训练层之外。

第一，环境要可扩展。你不能为每个任务手写一套脆弱流程。

第二，rollout 要撑得住。多轮工具调用慢，一慢训练就贵。

第三，奖励要可信。奖励不稳定，模型学到的就是噪声。

第四，任务要混得起来。不同任务的轨迹长度、奖励密度、动作空间差异太大，混合训练很容易偏向某一类任务。

第五，模型更新不能把会做的事情弄坏。Agent 训练有时会强化某些短期有效但长期危险的行为，比如过早停止、少验证、猜答案。

所以 AgentRL 的工程价值，不只是“用了 RL”。它更像是在给 Agentic RL 搭一个能规模化跑实验的底座。

我现在怎么理解 AgentRL

我会把 AgentRL 理解成一套“训练行动策略”的系统，而不是一套“提升回答质量”的技巧。

它真正关心的是：模型在一个会变化的环境里，能不能根据观察选择下一步动作，并通过可验证反馈慢慢形成更好的做事习惯。

基础概念里最重要的分界线，是回答和行动的区别。

回答只需要在当前上下文里生成文本。行动需要改变环境，再根据环境返回继续决策。

第一性原理也可以压成一句话：AgentRL 把 prompt-response 训练，改成 state-action-reward 训练。

这句话听起来简单，但它背后意味着整套系统都要变：数据不再只是样本，变成轨迹；标签不再只是答案，变成奖励；推理不再只是生成，变成和环境交互；训练瓶颈不再只是 GPU，也包括环境吞吐、工具稳定性和验证器质量。

以后再看类似框架，我会先看五件事：

  1. 它的环境接口是不是清楚。
  2. 它怎么做多轮 rollout。
  3. 它的 reward 是否真的可验证。
  4. 它怎么处理慢任务、坏轨迹和超时。
  5. 它最后提升的是“会说”，还是“会做”。

如果只提升最后回答，那还是偏对齐或推理优化。

如果它真的让模型在环境里更会行动，那才是 Agentic RL 该有的样子。
