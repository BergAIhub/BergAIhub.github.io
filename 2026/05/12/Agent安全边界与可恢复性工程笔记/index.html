Agent安全边界与可恢复性工程笔记

真正可用的 Agent，不是永远不犯错，而是犯错后不会把系统拖进不可逆状态。

先说结论

安全不是一个“额外模块”，而是 Agent 行为设计的骨架。

只做风险拦截，不做恢复设计，系统依然不可靠。

生产级 Agent 必须同时具备两种能力：避免高风险动作，且在失败后可回退、可诊断、可重试。

先把基础知识补齐

Agent 风险通常分三类：

  • 信息风险：泄露敏感数据，引用不可信内容。
  • 操作风险：执行破坏性命令，误改关键配置。
  • 决策风险：在证据不足时给出高置信结论。

可恢复性对应三种能力：

  • 状态可追踪：知道系统从哪个状态变到哪个状态。
  • 变更可回退：关键动作可撤销或重建。
  • 故障可归因：失败后能定位是工具、数据还是策略问题。

这个问题到底在讨论什么

我们不是在讨论“加几个 denylist 词”。

那只是输入过滤。

真正的问题是：Agent 在完整生命周期里，是否始终运行在受控边界内。

生命周期包括计划、执行、验证、交付、失败恢复。

任何一个环节失控，都会把前面所有防护抵消。

从基础往外延伸

常见安全设计分四层：

  • 预防层：权限最小化、工具白名单、动作分级。
  • 侦测层：异常行为监控、输出校验、敏感信息扫描。
  • 阻断层：高风险动作强制人工确认、自动熔断。
  • 恢复层：快照、回滚、重放、补偿事务。

很多系统只做前三层，忽略恢复层。

这会导致一个问题：你能阻止大部分事故，但一旦事故发生，就不知道怎么回家。

这个问题为什么容易想错

误区一是“把模型调谨慎一点就安全了”。

安全不是性格参数，而是系统约束。

误区二是“失败了再人工修”。

没有结构化日志和状态快照，人工修复会非常慢，且容易二次损伤。

误区三是“只要结果正确，过程不重要”。

在高风险场景里，过程可审计性和结果同等重要。

回到第一性原理

安全与恢复的底层约束来自不可逆性。

某些动作一旦执行，代价不是“多花点时间”，而是“数据或信任永久损失”。

因此系统必须在不可逆动作之前提高证据门槛。

同时承认另一个现实：错误必然发生。

所以恢复能力不是兜底，而是主流程的一部分。

真正的机制是什么

可靠 Agent 通常采用“分级权限 + 证据门控 + 事务化执行”：

  1. 分级权限：读、写、外部发布等动作分级，默认最低权限运行。
  2. 证据门控：高风险动作前要求多源证据或人工确认。
  3. 事务化执行：执行前记录快照，执行后验证状态，不通过则自动回滚或补偿。

这套机制把“是否安全”从主观判断变成工程流程。

用这个模型重新看问题

当你看到 Agent 偶发重大事故，通常不是偶发。

它意味着至少一个环节缺失：

  • 没有在高风险动作前抬高门槛。
  • 没有把动作和状态变更做可追踪绑定。
  • 没有预先定义失败后的自动恢复路径。

只修 Prompt，基本治标不治本。

我现在怎么判断

我会用六个问题做上线前检查：

  1. 是否为每类工具定义了最小必要权限。
  2. 是否标记了不可逆动作，并设置更高确认阈值。
  3. 是否记录了关键状态快照与变更日志。
  4. 失败后是否能自动回滚或执行补偿流程。
  5. 是否能把一次失败定位到具体环节和证据。
  6. 是否有演练过的故障处置手册，而不是纸面规则。

六条里缺两条以上，通常不建议直接上生产。

总结

安全的本质不是“永不出错”，而是“错误不会演变成灾难”。

可恢复性让 Agent 从一次性 Demo 变成可持续系统。

把边界和恢复写进架构，才能真正放心放权给 Agent。
