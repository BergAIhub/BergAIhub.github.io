Agent评测基线与回归门禁

没有评测基线，所有“变好了”都只是主观感觉。

先说结论

Agent 迭代速度快，退化也快。

回归门禁的目标不是阻碍上线，而是防止隐性退化进入生产。

先把基础知识补齐

评测至少要覆盖三类指标：

  • 任务成功率
  • 过程质量（步骤正确性、工具调用正确率）
  • 资源效率（延迟、成本、调用次数）

只看最终答对率，会掩盖过程层退化。

这个问题到底在讨论什么

不是做一套离线 benchmark 就结束。

核心是“每次改模型、改提示、改工具后，能否快速判定风险变化”。

从基础往外延伸

建议拆成两层：

  1. 离线回放：固定数据集，比较版本差异。
  2. 在线灰度：小流量真实环境，观察真实行为漂移。

离线告诉你“理论变化”，在线告诉你“现实变化”。

这个问题为什么容易想错

误区一：评测集长期不更新。

模型会对固定集合过拟合，指标好看但上线翻车。

误区二：只有平均值，没有分位数和长尾分析。

长尾失败往往才是用户投诉来源。

回到第一性原理

评测的本质是降低发布不确定性。

门禁规则要绑定业务风险，而不是追求学术完美分数。

真正的机制是什么

可执行门禁通常包含：

  • 硬门槛：关键指标低于阈值直接阻断。
  • 软门槛：轻微波动允许发布但触发告警。
  • 场景分桶：高风险场景单独评测，不被平均值淹没。
  • 回归归因：失败样本自动归类到模型/工具/数据。

我现在怎么判断

  1. 是否有稳定的黄金样本集和更新机制。
  2. 关键场景是否有独立门槛。
  3. 是否跟踪长尾失败率。
  4. 门禁失败是否给出可执行归因。
  5. 是否支持一键回滚到上个稳定版本。

总结

评测基线给你方向，回归门禁给你底线。

两者都在，Agent 才能既快迭代又不失控。
