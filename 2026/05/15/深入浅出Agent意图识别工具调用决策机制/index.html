深入浅出 Agent 意图识别工具调用决策机制

你说”帮我改个文件”，Agent 怎么知道该调 Write 还是 Edit？

用 Claude Code 写代码的时候，你随口说一句”把这个函数名改成 getUserInfo”，它就知道该用 Edit 工具做精确替换，不会用 Write 重写整个文件。你说”看看这个目录下有什么”，它选 Bash 跑 ls，不会用 Read 逐个打开文件。

这个决策过程看起来很自然，但背后有个问题值得拆开看：Agent 是怎么从一句自然语言里，判断出该调用哪个工具、传什么参数的？

传统做法是训练一个意图分类器，把用户输入映射到预定义的 intent 标签上，再用 slot filling 提取参数。Claude Code 没有这套东西。它的工具选择完全发生在 LLM 的推理过程中，没有独立的分类模块，没有 embedding 匹配，没有正则路由。

这篇文章用 Claude Code 作为唯一案例，拆解 Agent 意图识别的完整链路。

传统 NLU 意图分类和 LLM 工具调用是两件事

传统 NLU（Natural Language Understanding）的意图识别是一个分类问题：

  • 输入：用户的一句话
  • 输出：一个 intent 标签 + 若干 slot 值
  • 方法：训练一个分类器（BERT、BiLSTM-CRF 等），把输入映射到预定义的意图集合

比如智能客服里，”我要退货”映射到 intent: return_goods，”订单号是 123456”提取 slot: order_id = 123456。意图集合是固定的，新增一个意图就要重新训练模型。

LLM Agent 的工具调用不是这个模式。它没有预训练的意图分类头，没有固定的 intent 标签集。它的”意图识别”是在每次推理时，基于当前上下文（系统提示 + 工具描述 + 对话历史 + 用户消息）动态决定的。输出不是一个标签，是一个结构化的工具调用请求。

Claude 的 tool_use 协议：输入、输出、约束

Claude 的工具调用遵循一个明确的协议。

输入侧：工具定义

每个工具通过三个字段定义：

  • name：工具的唯一标识
  • description：告诉模型这个工具做什么、什么时候该用
  • input_schema：JSON Schema，定义参数的类型和约束

这三个字段就是模型做决策的全部信息来源。没有额外的路由逻辑，没有 embedding 索引。

输出侧：tool_use block

当模型决定调用工具时，响应里会包含一个 tool_use 内容块：

同时，响应的 stop_reason 会是 "tool_use" 而不是 "end_turn"。模型在告诉调用方：我需要执行一个工具，请把结果返回给我。

约束侧：tool_choice

开发者可以通过 tool_choice 参数控制模型的工具选择行为：

值 含义
auto 模型自己决定是否调用工具（默认）
any 必须调用某个工具，但哪个由模型决定
tool 强制调用指定的工具
none 禁止调用任何工具

auto 模式下，模型可能直接用文本回答而不调用工具。这是最接近”真实意图识别”的场景，模型需要判断：这个请求需要工具吗？需要哪个？

工具描述是意图识别的唯一信号源

模型选择工具的依据，就是工具的 description 字段。

Anthropic 官方文档建议 description 应该包含：

  • 工具做什么
  • 什么时候该用（什么时候不该用）
  • 每个参数的含义和对工具行为的影响
  • 重要的限制和注意事项

一个好的 description 至少 3-4 句话，复杂工具需要更多。来看 Claude Code 里 Edit 工具的描述（简化版）：

再看 Write 工具：

当用户说”把第 15 行的 foo 改成 bar”，模型看到 Edit 的描述里写着”exact string replacements”，就知道该用 Edit。当用户说”创建一个新文件 config.json”，模型看到 Write 的描述里写着”writing files”，就选 Write。

这不是关键词匹配，模型是在理解语义之后做的判断。如果你把 Edit 的 description 写成”A tool for modifying files”，模型在”改个函数名”和”重写整个文件”之间就会犯迷糊。

Claude Code 的工具体系：16+ 内置工具怎么分工

Claude Code 有 16 个以上的内置工具，每个都有明确的职责边界。模型需要在这些工具之间做出正确选择。

工具 职责
Read / ReadFile 读取文件内容
Write 创建或覆盖文件
Edit 精确字符串替换
Bash / PowerShell 执行 shell 命令
Grep 内容搜索（ripgrep）
Glob 文件名搜索
WebFetch 获取网页内容
WebSearch 搜索互联网
TodoWrite 任务列表管理
NotebookEdit Jupyter notebook 编辑
LSP 语言服务协议操作
Skill 元工具，加载和执行 skill
Agent/Task 子代理调度
AskUserQuestion 向用户提问

每个工具的 description 都经过精心设计，明确告诉模型”什么时候用我”和”什么时候别用我”。比如 Bash 工具的描述里会说：

这些”别用我”的指令，在帮模型做排除法。当用户说”搜索所有包含 TODO 的文件”，模型看到 Bash 说”别用我做 grep”，同时看到 Grep 说”content search using ripgrep”，就会选 Grep。

Skill 元工具：一个工具管理所有 Skill

Claude Code 有一个特殊的设计：Skill 不是直接暴露为独立工具，而是通过一个名为 “Skill” 的元工具统一管理。

这个 Skill 工具的 description 是动态生成的。每次 API 请求时，系统会把所有可用 skill 的名称和描述聚合成一段文本，塞进 Skill 工具的 description 里：

模型的 input_schema 只有一个字段：command（skill 名称）。

当用户说”帮我提取这个 PDF 的文字”，模型的决策过程是：

  1. 看到 Skill 工具的 description 里列着 "pdf": Extract text from PDF documents
  2. 判断用户意图匹配 pdf skill
  3. 输出 { "name": "Skill", "input": { "command": "pdf" } }

这里没有任何算法路由。没有 embedding 相似度计算，没有正则匹配，没有关键词检测。决策完全发生在模型的 forward pass 里。Han Lee 在他的源码分析中写得很明确：

The skill selection mechanism has no algorithmic routing or intent classification at the code level. This is pure LLM reasoning.

工具描述怎么进入模型上下文

当你调用 Claude API 并传入 tools 参数时，API 会自动构建一段特殊的系统提示：

工具定义被序列化成 JSON Schema 格式，拼接到系统提示里。模型看到的就是一段文本，工具名、描述、参数 schema 全部以文本形式呈现。

对于 Claude Code，每次对话时模型的上下文里都包含了所有 16+ 工具的完整描述，加起来可能占用数千 token。这也是 Anthropic 后来推出 Tool Search Tool 的原因，当工具数量膨胀到 50+ 时，全部塞进上下文会浪费 token 并降低选择准确率。

多步意图的拆解：复合请求怎么处理

用户经常一句话里包含多个意图。比如：”读一下 config.json，把 port 改成 8080，然后重启服务”。

Claude Code 处理这种情况的方式是顺序执行：

  1. 模型先输出第一个 tool_use（Read config.json）
  2. 你的代码执行工具，把结果作为 tool_result 返回
  3. 模型看到文件内容，输出第二个 tool_use（Edit，把 port 改成 8080）
  4. 执行，返回结果
  5. 模型输出第三个 tool_use（Bash，运行重启命令）

每一步都是一次完整的推理。模型在每一步都能看到之前的工具调用结果，基于最新的上下文做下一步决策。不是一次性规划好三步然后执行，每一步都是实时决策。

Claude 也支持在单次响应中输出多个 tool_use block（并行调用）。当模型判断两个工具调用之间没有依赖关系时，它可以同时发出多个请求。比如用户说”查一下北京和上海的天气”，模型可以同时调用两次 get_weather。

工具数量爆炸时的 Tool Search Tool

Claude Code 有 16 个内置工具，加上 MCP 服务器提供的工具，总数可能达到 50-100 个。把所有工具描述都塞进上下文有两个问题：

  1. Token 浪费，50 个工具的描述可能占 55K+ token
  2. 选择准确率下降，工具越多，名称相似的越多，模型越容易选错

Anthropic 的解决方案是 Tool Search Tool：

  • 大部分工具标记为 defer_loading: true，不加载到初始上下文
  • 只保留一个 Tool Search Tool（约 500 token）
  • 模型需要某个能力时，先搜索，再加载匹配的工具定义

内部测试显示，Opus 4 在大型工具库上的准确率从 49% 提升到 74%，Opus 4.5 从 79.5% 提升到 88.1%。

等于把意图识别拆成了两步：先识别大类（搜索），再在小范围内精确选择。

Claude Code 里一次完整的工具选择长什么样

拿一个真实场景：用户在 Claude Code 里说”帮我把 src/utils.ts 里的 getUserName 函数重命名为 getUsername”。

模型的上下文里有：

  • 系统提示（Claude Code 的身份和行为规则）
  • 16+ 工具的完整描述
  • 对话历史
  • 用户的这句话

模型的推理过程（我们看不到内部 reasoning，但可以推断）：

  1. 用户要重命名一个函数，涉及修改文件内容
  2. 排除不相关工具：不是搜索（排除 Grep/Glob），不是执行命令（排除 Bash），不是读文件（排除 Read）
  3. 在 Edit 和 Write 之间选择：Edit 的描述说”exact string replacements”，Write 说”writing files”。重命名是精确替换，选 Edit
  4. 构造参数：path = “src/utils.ts”，oldStr = “getUserName”，newStr = “getUsername”
  5. 输出 tool_use block

但这里有个细节：模型可能会先调用 Read 读取文件，确认函数确实存在、确认上下文足够做精确替换，然后再调用 Edit。因为 Edit 要求 oldStr 必须精确匹配文件内容，模型如果不确定文件内容，会先读一遍。

这个”先读后改”的行为不是硬编码的规则，是模型从工具描述中推断出来的。Edit 的描述里写着”oldStr must match EXACTLY”，模型理解了这个约束，所以在不确定时会先确认。

“没有算法路由”不等于”随机”

一个常见误解：既然没有算法路由，那工具选择是不是很随机？

不是。模型的决策受到多重约束：

工具描述的质量，描述越精确，选择越准确。Anthropic 建议每个工具至少 3-4 句描述。

负面指令，明确告诉模型”什么时候别用这个工具”，比如 Bash 描述里的那些 “prefer dedicated tools” 指令。

input_examples，提供具体的调用示例，帮助模型理解参数格式和使用模式。Anthropic 内部测试显示，加上 input_examples 后，复杂参数处理的准确率从 72% 提升到 90%。

工具命名，有意义的命名空间前缀（如 github_list_prs、slack_send_message）减少歧义。

Strict mode，grammar-constrained sampling 保证参数一定符合 schema，消除格式错误。

把 Agent 工具选择类比为意图分类，哪里不对

很多人把 Agent 的工具选择类比为意图分类。表面上看确实像：输入一句话，输出一个工具名。但有几个关键区别：

没有固定的标签集，工具可以动态增减，不需要重新训练。

参数生成是意图识别的一部分，传统 NLU 的 slot filling 是独立步骤，LLM 的工具选择和参数生成是同一次推理。

上下文敏感，同一句话在不同对话历史下可能触发不同工具。

可以选择不调用，模型可以判断”这个请求不需要工具，直接回答就行”。

还有一个容易忽略的点：模型可以在调用工具之前输出自然语言解释，比如”我来帮你查一下天气”然后跟一个 tool_use block。这段文本是模型推理过程中的副产物，不是意图识别的结果。

工具描述写得烂会怎样

工具描述质量直接决定意图识别的准确率。几个典型的失败模式：

描述太短：

模型不知道这是查数据库还是查搜索引擎，不知道什么时候该用。

描述有歧义：

发给谁？通过什么渠道？Slack？邮件？短信？模型只能猜。

多个工具描述重叠：

模型分不清这两个有什么区别。Anthropic 建议合并成一个工具，用 action 参数区分。

缺少负面指令：

如果 Bash 的描述里不写”prefer Grep for content search”，模型在搜索文件内容时就可能用 bash grep 而不是专用的 Grep 工具。

什么时候纯 LLM 推理做工具选择不够用

纯靠模型推理做工具选择，几个场景会遇到瓶颈：

工具数量超过 50 个，上下文里塞太多工具描述，模型注意力分散，准确率下降。这时需要 Tool Search Tool 做两阶段选择。

延迟敏感场景，每次工具选择都需要一次完整的模型推理（几百毫秒到几秒）。如果你的场景需要毫秒级响应，传统的规则路由或轻量分类器更合适。

确定性要求极高，模型的工具选择是概率性的，同一个输入可能偶尔选错。如果错误代价很高（比如金融交易），应该用 tool_choice: tool 强制指定，或者加一层确认机制。

工具调用需要编排逻辑，当一个任务需要循环调用工具（比如遍历 20 个文件做同样的操作），每次都走模型推理太慢太贵。这时 Programmatic Tool Calling 更合适，让模型写代码来编排工具调用。

以后我评估 Agent 工具系统会先看什么

工具描述的质量，不是看有多少工具，而是看每个工具的 description 写得够不够详细、够不够精确。描述是意图识别的唯一信号源，描述烂了什么都白搭。

工具之间的边界是否清晰，有没有两个工具的职责重叠？有没有明确的”什么时候别用我”指令？Claude Code 的 Bash 工具描述里那一堆”prefer dedicated tools”就是最佳实践。

规模化方案，工具数量超过 20 个以后怎么办？是全部塞进上下文，还是有按需加载的机制？Tool Search Tool 这种两阶段方案是目前的标准答案。

参数生成的可靠性，有没有 strict mode？有没有 input_examples？参数格式错误的处理机制是什么？

工具描述写得好，模型就能做出正确选择。很多人还在纠结”要不要加一个意图分类器”，大多数情况下不需要，把精力花在写好工具描述上，回报更高。
