文件 I/O 到底发生了什么：从 write() 到磁盘的一路

你调用 write()，它返回了成功。但数据真的到磁盘了吗？大概率没有。这篇文章从调用栈往下走，一直走到硬件，讲清楚操作系统文件 I/O 在每一层做了什么。

write() 返回 4096，看起来写完了。但此时如果拔电源，这 4096 字节大概率就丢了。它们只是被拷进了内核的 Page Cache，标记为”脏页”，还没落到盘上。

这不是 bug。这是操作系统故意为之的设计。理解这个设计为什么存在、每一层在做什么、以及你什么时候需要绕开它，就是这篇文章要讲的事。

一层一层往下走：从 VFS 到磁盘驱动

VFS：一个统一入口，对接几十种文件系统

VFS（Virtual File System）是内核里的一层抽象。它定义了四个核心结构：

  • superblock：描述一个已挂载的文件系统（块大小、设备信息、文件系统类型）。
  • inode：代表一个文件，存权限、大小、时间戳、数据块指针。不存文件名。
  • dentry：文件名到 inode 的映射。这是为什么 rename() 在同一文件系统内是 O(1)：只改 dentry，不动 inode 和数据。
  • file：进程打开文件时创建，持有当前读写偏移。

你写 open("/home/user/data.txt", ...) 时，内核做”路径行走”（path walking）：/ → home → user → data.txt，每一步先查 dentry 缓存（dcache），未命中才去磁盘读目录项。

ext4、XFS、NFS、tmpfs 都实现同一套 VFS 接口，上层不需要知道下面跑的是什么文件系统。

脏页什么时候真正写回？五个触发条件：

  • 脏页数量超过 dirty_ratio 阈值
  • 脏页驻留时间超过 dirty_expire_centisecs
  • 内存紧张需要回收页面
  • 显式调用 fsync() / sync()
  • 文件关闭

这就是为什么数据库的 fsync() 调用那么关键：在 fsync() 返回之前，数据是靠不住的。

Block 层和 IO 调度器

Page Cache 之下是 Block 层。BIO（Block I/O）请求在这里被合并、排序、派发。用 blk-mq（多队列）架构后，每个 CPU 核心有自己的提交队列，减少了锁竞争。

IO 调度器的选择取决于硬件：

调度器 适用场景
none/noop NVMe SSD，不需要排序，FIFO 即可
mq-deadline 数据库 + HDD，按 deadline 排序防饥饿
bfq 桌面交互，按进程公平分配带宽
kyber 低延迟设备，基于延迟反馈动态调整队列深度

缓冲 I/O、直接 I/O、同步 I/O：三件经常被搞混的事

这三组概念正交，但经常一起出现，混在一起说很容易搞错。

  • 缓冲 I/O（Buffered）：走 Page Cache。默认模式。
  • 直接 I/O（Direct，O_DIRECT）：绕过 Page Cache，用户缓冲区和块设备直接对接 DMA。
  • 同步 I/O（O_SYNC）：走 Page Cache 但等价于 write-through，每次 write 必须等磁盘确认才返回。

数据库喜欢 O_DIRECT：它们自己在用户态管理 buffer pool，再用一次 Page Cache 就是白白浪费内存和拷贝开销。代价是必须自己处理对齐（通常是 4KB 边界和扇区大小整数倍），而且失去了内核预读和 writeback 合并的好处。

零拷贝：目标是让 CPU 不用当搬运工

如果只是把文件数据发到网络，传统 read() + write() 的路径非常浪费：四次上下文切换、两次 CPU 参与的数据拷贝（内核和用户空间各一次）。数据根本没被用户态程序加工过，只是搬运，但 CPU 在搬这 4096 字节时就不能干别的。

sendfile() 把这件事收到内核里做：

数据路径变成：磁盘 →DMA→ Page Cache →DMA Gather→ 网卡。如果网卡支持 scatter-gather，CPU 拷贝次数是 0。Kafka 发消息用的就是这个。

splice() 更通用：在两个文件描述符之间通过管道在内核里直接移动页面引用，不拷贝数据本身。sendfile 只能文件→socket，splice 可以任意 fd↔pipe↔任意 fd。

选择零拷贝技术时记住一个教训：小于 4KB 的 payload，页引用计数开销可能比拷贝数据还大。

io_uring：把系统调用本身也省了

Linux 5.1 引入的 io_uring 改变了异步 I/O 的游戏规则。它用一对用户态和内核态共享的环形缓冲区（SQ 提交队列 + CQ 完成队列）来传递 I/O 请求：

  • 用户态往 SQ 里写 SQE（描述一个 read/write/send/recv 操作）
  • 内核处理后往 CQ 里写 CQE（完成结果）
  • 启用了 SQPOLL 模式后，内核线程持续轮询 SQ，用户态连 io_uring_enter() 都不需要调，零系统调用

4KB 随机读的实测数据（单 NVMe，队列深度 128）：

  • 传统 read/write：基线
  • libaio：~430K IOPS
  • io_uring 默认模式：~500K IOPS
  • io_uring + SQPOLL：~770K IOPS（接近 SPDK 用户态驱动的 ~950K）

io_uring 的代价是安全性。它支持 60 多种操作类型，攻击面很大。Google 安全团队在 2022 年发现 60% 的内核漏洞和 io_uring 有关。所以 Android、ChromeOS、Docker 默认容器都禁用了它。

抛开术语，文件 I/O 本质上是在管理三样东西

数据（数据本身在哪里）、位置（数据在文件中的偏移和磁盘上的物理位置之间的映射）、时机（数据什么时候从缓存刷到持久存储）。

Page Cache 解决的是时机：用延迟写换取吞吐。Direct I/O 放弃了这个优化，换取数据位置的完全控制权。零拷贝解决的是数据的搬运路径：让数据在内核里直接流转，不经过用户态。io_uring 解决的是时机的粒度：把单次系统调用变成批量异步提交。

以后我会先看它到底缓冲在哪一层

以后再看到一个存储系统说自己”高性能 I/O”，我会先问三个问题：

  1. 它走 Page Cache 还是 Direct I/O？如果走 Direct，它自己的缓存策略是什么？
  2. 它有没有调 fsync()？什么时候调？调完真的等了吗？
  3. 它的 I/O 模式是同步、异步、还是 io_uring？队列深度多少？

这三个问题的答案基本决定了它的性能上限和崩溃安全性。
