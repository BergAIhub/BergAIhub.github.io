操作系统进程线程协程的本质区别

发表于2026-05-13|更新于2026-05-13|Agent学习笔记
|浏览量:

进程、线程、协程：一次讲清楚它们到底在争什么

这篇文章从操作系统的视角出发，把进程、线程、协程三个概念拆到底层实现层面来讲。读完你会明白它们各自拥有什么资源、切换时付出什么代价、以及为什么现代语言运行时要在内核线程之上再造一层调度。

为什么这三个词总是一起出现却总是讲不清

学操作系统的时候，教材会告诉你：进程是资源分配的基本单位，线程是调度的基本单位。然后你去写 Go，发现 goroutine 既不是进程也不是传统线程；去写 Python，发现 asyncio 的协程跑在单线程里却能并发处理上千连接。

混乱的根源在于：这三个概念不在同一个抽象层。进程和线程是操作系统内核提供的抽象，协程是用户态运行时提供的抽象。它们解决的问题不同，付出的代价不同，能做到的事情也不同。

把它们放在一起比较，必须先搞清楚每一层到底管了什么。

Linux 内核眼里只有 task_struct

在 Linux 内核的实现里，根本没有”进程”和”线程”两种不同的数据结构。内核只有一种东西：task_struct。每一个可调度的执行单元，不管你在用户空间叫它进程还是线程，在内核里都是一个 task_struct 实例。

区别在哪？在于创建时传给 clone() 系统调用的标志位。

这些标志位决定了新创建的 task 和父 task 之间共享哪些资源：

标志位 含义 进程 线程
CLONE_VM 共享虚拟地址空间 ✗ ✓
CLONE_FS 共享文件系统信息（cwd、umask） ✗ ✓
CLONE_FILES 共享文件描述符表 ✗ ✓
CLONE_SIGHAND 共享信号处理器 ✗ ✓
CLONE_THREAD 同一线程组（共享 PID） ✗ ✓

所以 Linux 的设计哲学是：进程和线程不是两种东西，而是资源共享程度不同的同一种东西。Wellesley CS249 课程里有一句经典总结：”In the Linux kernel, there is no such thing as a thread.” 内核只看到 task，至于这个 task 是不是和别的 task 共享内存，那是创建时的配置问题。

进程拥有什么：一个完整的隔离世界

一个进程拥有的资源清单：

独立的虚拟地址空间。每个进程有自己的页表，虚拟地址 0x400000 在进程 A 和进程 B 里映射到不同的物理页。这是隔离的根基——一个进程的野指针不会写坏另一个进程的内存。

内存布局。代码段（.text）、数据段（.data/.bss）、堆（向上增长）、栈（向下增长）、内存映射区（mmap）。这些都在进程自己的虚拟地址空间里。

文件描述符表。进程打开的所有文件、socket、pipe 的引用。

信号处理表。每个信号对应的处理函数。

进程控制块（PCB）。在 Linux 里就是 task_struct，包含进程状态、优先级、调度信息、内存管理信息（mm_struct 指针）、文件信息（files_struct 指针）等。

进程切换时，内核需要做的事情：

  1. 保存当前进程的 CPU 寄存器状态（通用寄存器、程序计数器、栈指针、状态寄存器）。
  2. 切换页表基址寄存器（x86 上是 CR3）。
  3. 刷新 TLB（Translation Lookaside Buffer），因为新进程的虚拟地址映射完全不同。
  4. 恢复目标进程的寄存器状态。

其中第 3 步代价最大。TLB 是 CPU 里缓存”虚拟地址→物理地址”映射的硬件缓存，刷新后所有地址翻译都要重新走页表查找，直到 TLB 重新热起来。实测数据：在 x86_64 上，进程切换（不同地址空间）大约需要 1-5 微秒，而 TLB 冷启动带来的间接开销可能额外增加 5-15 微秒的缓存未命中延迟。

线程共享什么：同一屋檐下的独立执行流

同一进程内的线程共享：

  • 虚拟地址空间（堆、全局变量、代码段都是同一份）
  • 文件描述符表
  • 信号处理器
  • 进程 ID（对外表现为同一个进程）

每个线程独有的：

  • 独立的栈。每个线程有自己的调用栈，默认大小通常是 1-8 MB（Linux pthread 默认 8 MB）。
  • 独立的寄存器上下文。程序计数器、栈指针、通用寄存器。
  • 线程 ID（TID）。内核用来区分同一进程内不同线程的标识。
  • 线程本地存储（TLS）。通过 __thread 或 thread_local 声明的变量。
  • 信号掩码。每个线程可以独立屏蔽某些信号。

线程切换时，因为不需要切换页表（地址空间相同），所以：

  1. 保存当前线程的寄存器状态。
  2. 恢复目标线程的寄存器状态。
  3. 不需要刷新 TLB。

这就是线程切换比进程切换快的核心原因。实测数据：同进程内的线程切换大约 0.3-1.0 微秒，比进程切换快 3-10 倍。

但线程不是免费的。每个线程的栈空间（默认 8 MB 虚拟内存，实际按需分配物理页）、内核中的 task_struct（约 1.7 KB）、以及内核栈（通常 8-16 KB），加起来意味着一个系统能支撑的线程数量是有上限的。创建一万个线程就需要预留约 80 GB 的虚拟地址空间（虽然物理内存按需分配），内核调度器也会因为就绪队列过长而变慢。

协程不经过内核：用户态的执行流切换

协程（coroutine）的核心特征：切换不经过内核，完全在用户态完成。

操作系统内核不知道协程的存在。对内核来说，一个跑着一百万个协程的 Go 程序，可能只有几个线程在运行。协程的调度、切换、创建、销毁，全部由用户态的运行时（runtime）或库来管理。

协程切换需要保存和恢复的状态极少：

  • 程序计数器（下次从哪里继续执行）
  • 少量寄存器
  • 协程自己的栈（或者状态机的当前状态）

不需要：

  • 系统调用（不陷入内核）
  • 页表切换
  • TLB 刷新
  • 内核态/用户态的特权级切换

这使得协程切换的开销可以低到几十纳秒级别，比线程切换快 1-2 个数量级。

有栈协程和无栈协程：两条完全不同的路

协程的实现分为两大流派，它们的底层机制完全不同。

有栈协程（Stackful Coroutine）

代表：Go 的 goroutine、Lua 的 coroutine、早期 Java 的虚拟线程。

每个协程有自己的独立栈。切换时保存整个栈的上下文（寄存器 + 栈指针），跳转到另一个协程的栈继续执行。

Go 的 goroutine 是有栈协程的典型实现：

  • 初始栈大小只有 2 KB（对比 OS 线程的 8 MB）。
  • 栈可以动态增长：当函数调用发现栈空间不够时，runtime 会分配一个更大的栈，把旧栈内容复制过去。
  • 这使得一个进程可以轻松创建数十万甚至上百万个 goroutine。

有栈协程的优势：可以在任意调用深度挂起。你在第 10 层函数调用里做一个 channel 操作，整个调用链都会被挂起，恢复时从第 10 层继续。对程序员来说，写起来和同步代码一样自然。

有栈协程的代价：每个协程都要分配栈内存（即使很小），栈增长时需要复制，GC 需要扫描所有协程的栈。

无栈协程（Stackless Coroutine）

代表：Python 的 async/await、Rust 的 async/await、C++20 的 coroutine、JavaScript 的 async/await。

无栈协程没有独立的运行时栈。编译器（或解释器）把 async 函数转换成一个状态机。每个 await 点是一个状态转换边界，协程挂起时只保存当前状态编号和跨越 await 点的局部变量。

无栈协程的代价：只能在 await 点挂起（不能在任意调用深度挂起），需要 async 传染（调用 async 函数的函数也必须是 async 的），编写复杂递归或回调密集的异步逻辑时会比较痛苦。

抛开术语，它其实是在回答”谁来决定下一个执行谁”

把三者放在一起，核心区别可以用一个问题来统一：谁在做调度决策？

维度 进程 线程 协程
调度者 内核调度器 内核调度器 用户态运行时
切换方式 抢占式 抢占式 协作式（或半抢占式）
切换触发 时间片耗尽/IO阻塞/更高优先级任务就绪 同左 主动让出（yield/await）或运行时插入的检查点
切换代价 最高（TLB刷新+寄存器+内核态切换） 中等（寄存器+内核态切换） 最低（用户态寄存器/状态机切换）
并行能力 可以跑在不同 CPU 核上 可以跑在不同 CPU 核上 单个协程调度器内串行，需要多线程承载才能并行

抢占式意味着：你的代码正在执行，内核可以在任意指令边界打断你，把 CPU 给别人。你无法控制什么时候被切走。

协作式意味着：你的代码不主动让出，就不会被切走。这带来一个风险——如果一个协程执行了 CPU 密集的长计算而不 yield，其他协程就会饿死。

Go 的 goroutine 是个有趣的混合体：它本质上是协作式的（在函数调用、channel 操作、IO 操作时检查是否需要让出），但从 Go 1.14 开始加入了基于信号的抢占机制——runtime 会给长时间运行的 goroutine 发送信号，强制其让出。这使得 goroutine 在实践中表现得像抢占式调度。

Go 的 GMP 模型：用户态调度器的工程实现

Go 的运行时调度器是理解”为什么要在内核线程之上再造一层”的最佳案例。

GMP 模型的三个角色：

  • G（Goroutine）：用户态的执行单元。包含栈、程序计数器、状态等。
  • M（Machine）：操作系统线程。真正执行代码的载体。
  • P（Processor）：逻辑处理器。持有本地运行队列，连接 G 和 M。P 的数量默认等于 CPU 核数。

执行流程：

这个设计解决了什么问题？

  1. 线程数量可控。不管你创建多少 goroutine，实际的 OS 线程数量约等于 CPU 核数。避免了大量线程带来的内核调度开销和内存浪费。
  2. 系统调用不阻塞其他 goroutine。当一个 goroutine 陷入系统调用时，它所在的 M 会被阻塞，但 P 会转移到其他 M 上继续服务队列中的其他 goroutine。
  3. 负载均衡。work stealing 确保所有 P 都有活干，不会出现一个核忙死、其他核闲着的情况。

Python asyncio 的事件循环：单线程里的并发

Python 的 asyncio 是无栈协程 + 事件循环的典型实现。

核心组件：

  • 事件循环（Event Loop）：单线程运行，维护一个就绪队列和一个 IO 多路复用器（Linux 上是 epoll）。
  • 协程（Coroutine）：async def 定义的函数，调用后返回一个协程对象。
  • Task：对协程的封装，注册到事件循环中被调度。

执行机制：

当协程执行到 await 时：

  1. 协程把控制权交还给事件循环。
  2. 事件循环把这个协程关联的 IO 操作注册到 epoll。
  3. 事件循环继续执行其他就绪的协程。
  4. 当 IO 完成时，epoll 通知事件循环，协程被重新放入就绪队列。

这就是为什么 asyncio 能在单线程里并发处理上千连接——CPU 从不空等 IO，总是在执行某个协程的计算逻辑。但如果某个协程做了 CPU 密集计算（比如循环一百万次），整个事件循环都会被卡住，因为没有抢占机制。

容易搞混的几个点

“协程是轻量级线程”这个说法对不对？

不完全对。协程确实比线程轻量（内存占用小、切换快），但它们的调度模型根本不同。线程是抢占式的，协程是协作式的。线程可以真正并行（跑在不同核上），单个调度器内的协程只能并发（交替执行）。把协程叫”轻量级线程”会让人误以为它们是同一种东西的不同规格，实际上它们的行为语义有本质差异。

“多线程一定比多进程快”对不对？

不一定。线程切换确实比进程切换快，但多线程引入了共享内存的同步问题（锁、原子操作、内存屏障）。在某些场景下（比如 CPU 密集的独立计算），多进程反而更简单可靠——每个进程有独立的内存空间，不需要加锁，还能利用 COW（Copy-On-Write）减少 fork 的开销。Nginx 和 Redis 都选择了多进程/单线程模型。

“协程不能利用多核”对不对？

看实现。Python 的 asyncio 确实跑在单线程里，无法利用多核。但 Go 的 goroutine 通过 GMP 模型把协程分散到多个 OS 线程上执行，可以充分利用多核。所以准确的说法是：协程本身不直接对应 CPU 核，但运行时可以把协程调度到多个线程上实现并行。

进程间怎么通信？

既然进程有独立的地址空间，它们之间不能直接读写对方的内存。需要通过内核提供的 IPC 机制：管道（pipe）、消息队列、共享内存（需要显式映射同一块物理内存）、信号、socket。每种机制都有不同的性能特征和适用场景。

以后我判断该用哪个，就看这三件事

  1. 隔离需求。需要故障隔离（一个崩了不影响其他的）→ 进程。不需要 → 线程或协程。

  2. 并发量级。几十个并发 → 线程够用。成千上万个并发（典型的网络服务器场景）→ 协程，因为线程的内存和调度开销扛不住。

  3. 阻塞模型。如果并发单元主要在等 IO → 协程最合适（切换代价最低）。如果主要做 CPU 计算 → 线程或进程（需要真正的并行，协程的协作式调度反而是负担）。

一个实际的判断流程：Web 服务器处理 10000 个并发连接，每个连接大部分时间在等网络 IO → 用协程（Go/Node.js/Python asyncio）。视频转码需要吃满所有 CPU 核 → 用多线程或多进程。微服务之间需要进程级隔离防止一个服务的内存泄漏拖垮整个系统 → 用多进程。

没有银弹，只有 tradeoff。理解了每一层的代价和能力，选择就是自然的。
