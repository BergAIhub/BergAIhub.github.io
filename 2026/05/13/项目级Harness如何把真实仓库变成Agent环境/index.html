项目级 Harness 是怎么把一个真实仓库变成 Agent 环境的

评测从”做一道题”挪到”修一个仓库”以后发生了什么

两年前评测一个 coding agent，跑一遍 HumanEval、MBPP 就能出数。现在再拿这两个 benchmark 打榜 Claude Sonnet、GPT-5、DeepSeek-R1，全挤在 95% 以上，已经分不出差距。真正能拉开差距的是 SWE-bench Verified、Multi-SWE-bench、TerminalBench-2、Commit0 这类”项目级”任务：给你一个真实仓库、一条 issue，让你自己去看代码、改代码、跑测试、提交 patch。

这类任务跑起来立刻会碰到一个具体的问题：怎么跑。一条 issue 对应的 repo 依赖十几个 pip 包，某个 conftest 还要求系统层装了 clang，测试套件要跑二十分钟，CI 里还藏着一个只有在 GitHub Actions runner 上才触发的环境变量。每次评测都手搓一遍这套环境是不现实的。

这层”把真实 repo 变成 agent 可交互、可判分的环境”的工程装置，就是 project-level harness。它决定了评测能不能复现、训练能不能 scale、数据会不会被污染。表面是工程细节，实际上 Meta-Harness 那篇论文测出来：同一个模型只换 harness，性能差 6 倍。harness 不是 benchmark 的附属品，是 benchmark 本身。

下面把项目级 harness 拆开讲，尽量讲清楚 SWE-bench、R2E-Gym、Commit0、Multi-SWE-bench、OpenHands、Meta-Harness 这几条路线各自在解决什么问题，再回到”自己要给一个项目搭一套 harness 该怎么做”。

先把 harness、benchmark、environment 这三个词分开

这三个词在 agent 圈几乎混用，但在工程上是三件不同的事。

benchmark 是一批任务实例加一套判分规则。SWE-bench Verified 就是 500 条人工 review 过的 GitHub issue，判分规则是”patch 能让 FAIL_TO_PASS 的测试翻过来且不引入 regression”。它是一个数据集。

environment 是一个可执行的仓库快照。一个 base commit 加一套能让仓库跑起来的依赖和运行时，让 agent 在里面发命令、改文件、跑测试。它是一台装好软件的机器。

harness 是把 benchmark 跑在 environment 上的那套代码。它读取一条 task instance、拉起对应 environment（一般是一个 Docker 容器）、把 agent 接进去、收集 patch、应用 patch、跑测试、按 benchmark 规则打分、清理资源、并发下一条。它是把数据集和机器串起来的流水线。

SWE-bench 官方仓库里 swebench.harness.run_evaluation 就是字面意义上的 harness 模块；OpenHands 的 openhands/runtime/impl/docker/docker_runtime.py 是另一个层面的 harness，偏 agent 执行端。三者缺一不可，但研究里经常只谈 benchmark 不谈 harness，于是评测跑不出一致的数。

一条 task instance 里到底有哪些字段

两组字段值得单独说。

一是 base_commit 和 environment_setup_commit 通常不是同一个 commit。真实 repo 的依赖列表（package.json、pyproject.toml）会随时间漂移，完全按 issue 发生时的 commit 装依赖，可能跑不起来。SWE-bench 的做法是挑一个该 repo 在那段时间最稳定的 commit 当 environment_setup_commit，用它装 Python 版本和 pip 依赖，再 checkout 到 base_commit 去应用 patch 跑测试。

二是 FAIL_TO_PASS 和 PASS_TO_PASS 不是测试集合，是行为约束。前者是”这些测试在没 fix 时一定失败，打上 fix 后必须通过”，后者是”这些测试在 fix 前就在通过，fix 后必须还通过”。后一组就是 regression check。这两组是整个 Oracle 的骨架，下面详细讲。

三层镜像是 SWE-bench 做出来的空间换时间

最朴素的做法是每条 instance 建一个独立 Docker 镜像，从 python:3.10-slim 起装 apt 包、装 pip 依赖、clone repo、checkout commit。SWE-bench Lite 有 300 条 instance，每条镜像 3-5GB，全量建完 1-2TB，每条单独走一遍会重复装几十次相同的 Python 3.10 和相同的 django、numpy、pytest。

SWE-bench 官方 harness 把镜像拆成三层：

base image 全 benchmark 共用几个；environment image 每个 repo 每个版本一个；instance image 每条 instance 一个，但它只是在 environment image 上再加 git checkout 和 git apply --check 这种轻动作。SWE-bench harness 的 --cache_level 参数直接控制缓存留在哪一层：

cache_level 保留范围 磁盘 重建速度
none 不缓存 运行时 ~120GB 最慢
base 只缓存 base 运行时 ~120GB 慢
env（默认） 缓存 base + env ~100GB 中
instance 缓存全部 ~2,000GB 最快

除非反复跑同一个预测文件，env 级够用。这也是为什么官方文档写”多数用户用默认值”。

OpenHands 走的是另一条分层思路。它的 runtime image 用三层 tag 系统（versioned tag / lock tag / source tag），根据 pyproject.toml 的哈希和源目录哈希决定要不要重建。命中最高层的 source tag 就完全跳过构建；命中 lock tag 就复用依赖层，只拷源码。目的一样：让热路径尽量命中缓存。

run.log、test.log、fix.log 三路对照是 Oracle 的骨架

判分是整个 harness 最容易被低估、也最容易出错的部分。SWE-bench 和 Multi-SWE-bench 用的是三路对照，这个方法值得每个搭 harness 的人记一遍。

对每条 instance，准备三份测试运行结果：

每个 test case 会拿到一个状态三元组，比如 PASSED → FAILED → PASSED。Multi-SWE-bench 的筛选规则是：

  • 存在至少一个 ANY → FAILED → PASSED 的 case（bug 确实被修好）
  • 不存在任何 ANY → PASSED → FAILED 的 case（没引入 regression）
  • 排除 PASSED → NONE/SKIPPED → FAILED 这种异常 transition

同时满足前两条的 instance 才能进 benchmark。评测时 agent 产生一个候选 patch，用同样三路跑一遍，和 gold fix 的结果对齐得出 pass。

Multi-SWE-bench 的 troubleshooting 章节记了一堆踩坑，几条特别能说明 Oracle 为什么难：

  • pre-fix 阶段有些 repo 直接编译不过，因为 test_patch 里引用了 fix 才创建的函数
  • C/C++ 会把 .o、.bin 留在工作目录，git apply 报错，要硬编码过滤
  • Java 并行测试的 stdout 交错在一起没有分隔符，rule-based parser 直接失败
  • 部分测试名带时间戳或随机后缀，每次运行 test_id 都不一样，必须排除

这些不是实现细节，是 Oracle 设计能不能用的分水岭。自己搭 harness 如果不把三路对照写进流水线，拿到的 pass rate 很可能是错的。

Action server 的 client-server 结构让 agent 能长寿命交互

早期很多 agent 评测是这样做的：把整个 agent loop 塞进一个 Docker 容器，里面直接 subprocess.run("pytest")。问题在于：

  • agent 崩了容器也就没了，没法 resume
  • shell 是无状态的，cd somedir && export X=1 && pytest 这种会话状态传不下来
  • 没法同时跑 shell、Python REPL、浏览器、代码编辑器

OpenHands 的 Runtime 的解法是把 agent 和执行环境分成两个进程：

backend 跑在宿主机做 LLM 推理；容器里跑一个 action_execution_server.py，暴露一组 REST endpoint 处理 bash、file_edit、ipython_run 这些动作。shell 用 pexpect 包一个持久化 bash session，cd 和 export 的状态能留住；Jupyter 用 Kernel Gateway 起一个持久 kernel；编辑器挂 VSCode server，用 token 给 web URL。

这套结构的价值是 agent 和环境解耦。backend 可以部署在 A100 上跑 70B 模型，容器可以部署在一批便宜 CPU 机器上，靠端口锁（find_available_port_with_lock）和 DOCKER_HOST_ADDR 管并行。SWE-agent 自己另外做了一个叫 SWE-ReX 的远程执行框架，解决的是同一类问题：agent 要能管一堆长寿命 shell session，而不是一个命令开一个容器。

Verifier 分两大类，混用才跑得稳

Verifier 就是判分器，SWE-bench 那条”跑 FAIL_TO_PASS 测试”只是最传统的一种。目前前沿分两条路。

Execution-based verifier：最直接，跑测试看通过率。信号硬，但 R2E-Gym 那篇论文测出来：实际有效的 distinguishability 只有 20% 左右。意思是一条 issue 上 agent 生成 10 个候选 patch，往往只有少数测试能区分”对的”和”错的”，多数测试对所有候选都给一样的结果（全过或全挂）。更麻烦的是 toxicity：错的 patch 能通过，对的 patch 反而挂，通常是因为测试本身写得有歧义。

Execution-free verifier：不跑代码，让一个训好的 LLM 判别器看 patch 内容和 agent trajectory 打分。细粒度、成本低。但 R2E-Gym 发现它的注意力严重依赖 agent 的”思考过程”：trajectory 里 agent 用了”Great”、”Successfully”、”the fix works” 这种自信措辞，verifier 倾向判它对，哪怕 patch 其实是错的。去掉 thoughts 只保留 patch 的 ablation 准确率掉 3-5%，说明 verifier 过分依赖启发式。

R2E-Gym 的做法是混合：先用 execution-based 过滤掉跑不过测试的死候选，再用 execution-free 对活下来的候选做精细排序。在 SWE-bench Verified 上，这种 hybrid 方案让开源 32B 模型的 pass@1 从 34% 提到 51%，只要 26 个 rollout，而不是以前工作里的 500+。

混合 verifier 不只是多跑一层。它等于承认：没有任何单一判分信号够用，harness 的 Oracle 层必须是可组合的。

评测 harness 和训练 harness 是同一套东西，但代价不同

SWE-Gym 的贡献是第一次把 SWE-bench 风格的环境用作 RL 训练：2,438 条 Python task，每条都带可跑 runtime 和 unit test。Multi-SWE-bench 发布时同步开了 Multi-SWE-RL 社区，放了 4,723 条跨 7 语言的 containerized instance 给 RL 社区用。逻辑很简单：评测时 harness 吐出的就是 reward，训练时也要 reward。

但真把评测 harness 当训练环境用会立刻出一堆新问题：

  • 评测允许单次串行跑几分钟，训练要上千并发，container cold start 扛不住
  • 评测只关心最终 pass/fail，训练要每一步 action 的中间观察、记 trajectory
  • 评测一条 instance 跑一次就行，训练要反复 reset 到 base_commit 干净状态
  • flaky test 在评测里是个 bug，在训练里是灾难，reward 噪声直接让 policy gradient 爆炸

NVIDIA NeMo Gym 的 SWE RL 工程笔记里写过他们怎么解决：用 Modal/K8s 的远程沙箱池，每个 rollout 租一个 worker；用 overlay filesystem 做 copy-on-write，reset 成本从”重建镜像”变成”丢 overlay 层”；用预运行过滤（pre-run filter）在训练开始前把 flaky 的 instance 全剔掉。

R2E-Gym 走的是另一条路：procedurally 生成 environment。它的 SWE-Gen 流水线不用人写 issue，直接从 commit 出发，用 LLM 生成 reproduction test，再 back-translation 生成自然语言 issue。这样能从几个 repo 生成 8,100 条 executable environment，成本比 Multi-SWE-bench 那种 68 个 annotator 手工 review 一年要低几个数量级。代价是合成任务和真实任务分布不完全一样，所以他们做 SWE-Bench Verified 测试时还是要在真实任务上验证。

抽一层看，项目级 harness 就是把 “issue → patch” 映射成可判定的 reward

把前面那些东西抽象一层，项目级 harness 做的事可以写成一个函数：

输入是一个 repo 的某个历史状态和一条 issue；agent 是能在环境里发动作的 policy；输出是这次尝试是否 pass。

中间的”状态”就是 Docker 容器的文件系统和进程表，”约束”是测试必须在 fix 前失败、fix 后通过且不 regression，”feedback”是每一步 action 的 stdout/stderr/文件变更/测试输出。

这么看项目级 harness 其实不特殊，它就是一个 RL environment，只是：

  • state 空间是整个文件系统（上 GB 级）
  • action 空间是任意 shell 命令（无限）
  • reward 很稀疏（只在 submit 时给）
  • episode 很长（20-50 turn）
  • 每一步都涉及真实 side effect（改了 repo 的状态）

所有工程复杂度都从这四条来。三层镜像解决 state 太大，REST 架构解决 action 多样，三路对照解决 reward 难定义，overlay filesystem 解决 side effect 难 rollback。

几个细节值得单独提：

  • max_workers 官方建议 min(0.75 * cpu_count, 24)。再大会因为 Docker daemon 锁争抢反而更慢。16 核机器并行 12 个 worker，跑 Lite 大概半小时。
  • 测试超时按 instance 设置（--timeout），默认 30 分钟，超时算失败。
  • --modal true 可以把整个流程跑在 Modal 云上，sb-cli 工具也能跑在 AWS。真实场景里团队常选一套 serverless container 平台分担 CI 压力，本地只留一台机器做开发。

Multi-SWE-bench 在这基础上多了两件事：一是每种语言有独立 test runner（go test、cargo test、mvn test、npm test），harness 要识别语言后路由；二是 .gitignore 要排除 .o、.bin 等编译产物，否则 git apply 会失败。

关于 harness 最常见的几个误解

“harness 就是 Docker 跑测试”。不是。Docker 只解决了”跑”的问题；harness 的难点在 Oracle 设计（三路对照）、依赖解析（从 CI 抽）、状态管理（overlay）、并发协调（端口锁 + cpu 限制）、判分规则（FAIL_TO_PASS / PASS_TO_PASS）。去掉这些只剩 Docker，那叫 Dockerfile。

“benchmark 测分越高模型越强”。Meta-Harness 那篇论文直接反例了这个假设：同一个 Claude Opus 4.6，配不同 harness，TerminalBench-2 pass rate 从 58%（Claude Code）跳到 76.4%（Meta-Harness 自己搜出来的 harness）。比较两个模型时没报告 harness，这个对比几乎没意义。

“数据集一旦发布就稳了”。SWE-rebench、SWE-bench-Live、SWE-MERA 这三个工作都在说同一件事：数据污染让静态 benchmark 的寿命越来越短。2024 年发布的 SWE-bench Verified 到 2025 年已经能被几个模型明显看出”见过”。SWE-rebench 用自动化流水线持续从 GitHub 抽新 issue 生成新 instance，每批任务都标注创建时间，评测时红色高亮可能被训练数据覆盖的条目。自己搭 harness 时记得把”时间戳 + 污染追踪”加进去。

“一种语言跑通就等于适配多语言”。Multi-SWE-bench 适配 SWE-agent、Agentless、OpenHands 到 7 种语言时发现每个方法都要大改：MagentLess 因为某些语言 file skeleton 抽不出来，要全文件输入；MSWE-agent 要加 .gitignore 过滤编译产物；MopenHands 要修 git diff 里 tab 被渲染成空格的 bug。多语言不是”加一个解释器”，是”把整套 tool interface 再审一遍”。

“harness 只跟评测有关，跟训练没关系”。正相反，现在大部分 agent 训练的核心工程就是 harness。SWE-Gym、Multi-SWE-RL、NeMo Gym、R2E-Gym 全是在搭训练用 harness。模型越来越便宜，训练环境越来越贵，harness 的杠杆越来越大。

什么时候不该自己造一套项目级 harness

这套东西工程量不小，也不是每个项目都值得搭。几种情况我会劝退：

repo 没有自动化测试。harness 的核心是 Oracle，Oracle 的核心是测试。项目本身没有 CI、没有 pytest/junit/go test，先去加测试，不是先搭 harness。

核心价值在 GUI 或者硬件。SWE-bench Multimodal 和 WebArena 在做这类任务，但它们的 harness 已经不是 Docker 文件系统这层了——需要 VM、模拟键鼠、截图 diff。搭这类 harness 的成本比普通项目级高一个数量级，除非你在做 UI agent 研究，否则不要自己从零做。

模型只需要评测几十次。手工评测可能比自动化 harness 更快。见过有团队花两周搭了一套完整 harness 去跑 30 条 case，最后被一个实习生用脚本加人工 review 半天干完。搭 harness 的损益平衡点大概在几百到上千次评测规模；低于这个数不如人工。

项目闭源、依赖大量私有服务。这类项目你没办法公开发布 environment，也很难让外部评测信任你的 harness。内部私有 harness 是可以的（Amazon SWE-PolyBench 就是私有版 benchmark 发 paper 时才开源），但需要评估维护成本值不值。

更好的替代：直接复用现成 harness。SWE-bench 官方仓库有完整的 “How to add a new dataset” 教程，把你项目的 instance 组织成符合 SWE-bench schema 的 jsonl，就能直接复用现成 harness。比自造轮子省一两个月工作量。

以后再看一个 agent 项目我会先翻它的 harness

写完这篇我对 agent 项目的评估方法变了。以前拿到一个项目先看模型是什么、用了什么 prompt 技术、有没有 tool use 的新 trick。现在我先翻 evaluation 目录，看三件事：

一，Oracle 是几路对照。只有一路 pytest 过不过那种我会怀疑结果有水分，因为 regression check 没做。

二，镜像是三层还是单层。单层意味着每次评测都要重建依赖，说明作者没跑过真正的全量评测，只是采样几条跑通。

三，harness 版本号是不是跟 benchmark 版本号绑定。像 SWE-bench 那样 --cache_level env 加 --version 5.0 的才是工程上稳的；只写 python eval.py 没版本的那种，隔半年复现会翻车。

Meta-Harness 那篇论文的结论让我印象很深：harness 本身是可以被搜索、被训练、被优化的。未来一两年里最有意思的方向未必是”再造一个新 agent”，而是”给现有 agent 配一套更好的 harness”。前者研究模型，后者研究模型和环境的接口。后者的杠杆更大，也更具体。
