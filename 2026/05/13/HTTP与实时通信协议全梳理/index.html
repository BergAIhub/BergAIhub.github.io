HTTP与实时通信协议全梳理

发表于2026-05-13|更新于2026-05-13|Agent学习笔记
|浏览量:

从一次请求到持久连接：HTTP、HTTPS、SSE、WebSocket 各自解决了什么问题

这篇文章把 HTTP、HTTPS、SSE、WebSocket 四个协议放在一起，从连接模型、数据流方向、协议开销三个维度拆开讲。读完你会清楚每个协议在网络栈里的位置、它们之间的演进关系、以及在实际场景中该选哪个。

一个前端页面为什么需要四种不同的通信方式

打开一个现代 Web 应用，你的浏览器可能同时在做这些事：

  • 用 HTTP GET 拉取页面 HTML、CSS、JS 和图片
  • 通过 HTTPS 向后端 API 发送带认证信息的请求
  • 用 SSE 接收 AI 模型逐 token 生成的回答（ChatGPT 的打字机效果）
  • 用 WebSocket 维持一个在线协作文档的双向实时同步

这四种通信方式不是互相替代的关系。HTTP 管”怎么请求和响应”，HTTPS 管”怎么加密”，SSE 管”服务器怎么持续推送”，WebSocket 管”双方怎么随时互发消息”。

搞混它们的人，通常是没分清两件事：连接的生命周期和数据流的方向。

HTTP 的本质：一问一答的文本协议

HTTP（HyperText Transfer Protocol）是应用层协议，跑在 TCP 之上。核心模型很简单：客户端发一个请求，服务器回一个响应，完事。

HTTP/1.1 的连接模型

HTTP/1.0 每次请求都要新建 TCP 连接（三次握手），用完就关。HTTP/1.1 引入了 Connection: keep-alive，允许在同一个 TCP 连接上串行发送多个请求。但关键限制是：队头阻塞（Head-of-Line Blocking）——前一个请求的响应没回来，后面的请求只能排队等。

HTTP 是无状态的——每个请求独立，服务器不记得上一个请求是谁发的。Cookie、Session、Token 都是在这个无状态协议之上叠加的状态管理机制。

HTTP/2：二进制帧 + 多路复用

HTTP/2（RFC 9113，2022 年正式标准化）解决了 HTTP/1.1 的队头阻塞问题。核心改进：

二进制分帧。HTTP/1.1 是纯文本协议（人类可读），HTTP/2 把所有通信拆成二进制帧（Frame），每个帧属于一个流（Stream）。

多路复用。同一个 TCP 连接上可以同时跑多个流，每个流是一个独立的请求-响应对。流之间互不阻塞——流 1 的响应还没回来，流 2 的响应可以先到。

头部压缩（HPACK）。HTTP 头部有大量重复内容（Host、Cookie、User-Agent），HPACK 用静态表 + 动态表 + 霍夫曼编码压缩头部，减少传输量。

服务器推送。服务器可以主动推送客户端还没请求但可能需要的资源（比如 HTML 里引用的 CSS）。但实践中这个特性用得不多，Chrome 已经移除了对 HTTP/2 Server Push 的支持。

HTTP/2 的局限：它仍然跑在 TCP 上。TCP 层的丢包会阻塞整个连接上的所有流（TCP 层的队头阻塞）。

HTTP/3：QUIC + UDP

HTTP/3（RFC 9114，2022）把传输层从 TCP 换成了 QUIC（跑在 UDP 上）。QUIC 的每个流独立处理丢包重传，一个流丢包不影响其他流。同时 QUIC 内置了 TLS 1.3，握手只需要 1-RTT（首次连接）或 0-RTT（恢复连接）。

HTTPS 不是一个新协议，是 HTTP + TLS 的组合

HTTPS = HTTP over TLS。它不改变 HTTP 的语义（请求方法、状态码、头部都一样），只是在 HTTP 和 TCP 之间插入了一层 TLS 加密。

证书验证做了什么

  1. 服务器发送证书链（服务器证书 → 中间 CA 证书 → 根 CA 证书）。
  2. 客户端用本地信任的根 CA 公钥验证证书链的签名。
  3. 检查证书是否过期、域名是否匹配、是否被吊销（OCSP/CRL）。
  4. 验证通过后，用证书中的公钥参与密钥交换，生成对称加密的会话密钥。

之后所有 HTTP 数据都用这个会话密钥做对称加密（AES-GCM 或 ChaCha20-Poly1305），性能开销很小。

HTTPS 保护了什么，没保护什么

保护了：

  • 请求和响应的内容（包括 URL 路径、头部、body）
  • 数据完整性（篡改会被检测到）
  • 服务器身份认证（防止连到假服务器）

没保护：

  • 目标 IP 地址（网络层可见）
  • DNS 查询（除非用 DoH/DoT）
  • 连接的元数据（时间、数据量、连接频率）
  • SNI 字段中的域名（TLS 握手时明文发送，除非用 ECH）

SSE：HTTP 长连接上的服务器单向推送

Server-Sent Events（SSE）是 HTML5 标准的一部分，它复用 HTTP 协议实现服务器向客户端的持续推送。

SSE 的连接模型

SSE 本质上就是一个不关闭的 HTTP 响应。服务器发送 Content-Type: text/event-stream，然后持续往响应体里写数据，连接一直保持打开。

事件流格式

SSE 的数据格式极其简单，纯文本：

每个事件由若干字段组成，字段之间用换行分隔，事件之间用空行（\n\n）分隔：

  • data: — 事件数据（可以多行）
  • event: — 事件类型（可选，默认是 message）
  • id: — 事件 ID（用于断线重连时告诉服务器从哪里继续）
  • retry: — 重连间隔（毫秒）

浏览器端的 EventSource API

EventSource 的内置特性：

  • 自动重连。连接断开后浏览器自动重新发起请求，并在请求头中带上 Last-Event-ID，服务器可以据此恢复推送。
  • 事件类型分发。可以用 addEventListener('custom-event', handler) 监听特定类型的事件。

为什么 AI 流式输出都用 SSE

OpenAI、Anthropic、Google 的流式 API 全部选择了 SSE 而不是 WebSocket。原因：

  1. 单向推送完美匹配。AI 生成是服务器单向输出 token，客户端只需要接收。
  2. HTTP 原生。SSE 走标准 HTTP，能过 CDN、负载均衡器、反向代理，不需要特殊配置。
  3. 自动重连。网络抖动时 EventSource 自动恢复，配合 id 字段可以断点续传。
  4. 简单。不需要维护连接状态，服务端就是往 response 里写文本。

WebSocket：升级到全双工的独立协议

WebSocket（RFC 6455，2011）是一个独立的应用层协议，它借用 HTTP 完成握手，然后切换到自己的二进制帧协议。

握手过程：从 HTTP 升级

握手完成后，这个 TCP 连接不再是 HTTP 连接了——它变成了 WebSocket 连接，使用完全不同的帧格式。

Sec-WebSocket-Key 和 Sec-WebSocket-Accept 的作用不是加密，而是防止缓存代理误把 WebSocket 流量当成普通 HTTP 缓存。服务器把客户端发来的 Key 拼接一个固定 GUID，做 SHA-1 哈希后 Base64 编码返回，证明它确实理解 WebSocket 协议。

帧格式

WebSocket 的数据以帧（Frame）为单位传输，帧头最小只有 2 字节：

关键字段：

  • FIN：是否是消息的最后一帧（大消息可以分片）
  • opcode：帧类型
    • 0x1 = 文本帧
    • 0x2 = 二进制帧
    • 0x8 = 关闭连接
    • 0x9 = Ping
    • 0xA = Pong
  • MASK：客户端发给服务器的帧必须掩码（防止缓存投毒攻击），服务器发给客户端的不需要
  • Payload len：7 位表示 0-125 字节，126 表示后续 2 字节是实际长度，127 表示后续 8 字节是实际长度

对比 HTTP 的文本头部（几百字节起步），WebSocket 帧头只有 2-14 字节，这就是为什么它适合高频小消息场景。

Ping/Pong 心跳

WebSocket 内置了心跳机制。任何一方都可以发送 Ping 帧，对方必须回复 Pong 帧。这用来：

  • 检测连接是否还活着
  • 防止中间代理因为空闲超时关闭连接

抛开协议名字，它们其实在回答”数据怎么流动”

用第一性原理来看，网络通信只有三个基本问题：

1. 谁先说话？

  • HTTP/HTTPS：永远是客户端先说（发请求），服务器只能被动回答。
  • SSE：客户端先发一个请求打开通道，之后服务器持续说话。
  • WebSocket：握手后双方随时可以说话，没有主从之分。

2. 说完一句话连接还在不在？

  • HTTP/1.0：说完就断。
  • HTTP/1.1 keep-alive：说完不断，但要排队等。
  • HTTP/2：说完不断，可以同时说多句。
  • SSE：服务器一直在说，直到主动停止或连接断开。
  • WebSocket：连接一直在，直到任何一方主动关闭。

3. 每句话的信封有多大？

  • HTTP：每句话都要带完整的头部（Host、Cookie、Content-Type…），几百字节起步。
  • SSE：只需要 data: 内容\n\n，十几字节。
  • WebSocket：2-14 字节的帧头。

这三个问题的答案组合，决定了每个协议适合什么场景。

实际选型：什么场景用什么

普通 API 调用 → HTTP/HTTPS

请求-响应模型，无状态，RESTful。绝大多数 Web 应用的主力通信方式。

AI 流式输出、实时通知、股票行情推送 → SSE

服务器单向推送，客户端只接收。SSE 的优势是简单、HTTP 原生、自动重连、CDN 友好。ChatGPT、Claude、Gemini 的流式 API 全部用 SSE。

在线聊天、协作编辑、多人游戏 → WebSocket

双方都需要随时发消息，且消息频率高、延迟敏感。WebSocket 的帧开销最小，全双工，适合高频双向通信。

文件上传/下载 → HTTP/HTTPS

大块数据传输，HTTP 的 chunked transfer 和 range request 已经很成熟。

一个容易犯的错误

很多人看到”实时”就想用 WebSocket，但实际上大部分”实时”场景是服务器单向推送——通知、消息提醒、数据更新、AI 输出。这些场景用 SSE 就够了，比 WebSocket 简单得多：

  • 不需要维护连接状态
  • 不需要处理双向消息的序列化
  • 不需要实现心跳和重连逻辑（EventSource 内置）
  • 能过 CDN 和标准 HTTP 基础设施

只有当客户端也需要高频向服务器发送数据（不是偶尔发一个 HTTP 请求，而是持续的双向数据流）时，WebSocket 才是必要的。

以后我判断用哪个协议，就看这两步

第一步：数据流方向。

  • 客户端请求 → 服务器响应（一问一答）→ HTTP/HTTPS
  • 服务器持续推送 → 客户端接收 → SSE
  • 双方随时互发 → WebSocket

第二步：确认约束条件。

  • 需要加密 → 加 TLS（HTTPS / wss://）
  • 需要过 CDN / 负载均衡 → 优先 SSE（HTTP 原生）
  • 消息频率极高（每秒几十上百条）+ 双向 → WebSocket
  • 需要自动重连和断点续传 → SSE 内置支持

别为了”看起来高级”选 WebSocket。多数场景 SSE 更简单、更可靠、更好运维。WebSocket 是给真正需要全双工的场景准备的——用对了是利器，用错了是负担。
